# Plan: `eco-boot` — Native Compiler Binary

## Goal

Create a C++ binary `eco-boot` that compiles Elm source files (or `.mlir` files directly) to standalone, fully statically linked x86-64 Linux ELF executables. It orchestrates:
1. The Elm compiler frontend (eco-boot-2.js running under Node.js) to produce `.mlir`
2. The MLIR→LLVM lowering pipeline (already exists in `ecoc`)
3. **New**: LLVM IR → object file emission via `TargetMachine::emit()`
4. **New**: Linking the object file with runtime/kernel static libraries into a final ELF binary

The output binary is fully self-contained — it statically links the Eco GC runtime, Elm kernel C++, Eco kernel C++, libc, libstdc++, and libpthread.

## Decisions

| Decision | Choice |
|---|---|
| Accept `.mlir` input directly? | Yes — detect by file extension, skip frontend for `.mlir` |
| Linking strategy | `clang++` as linker driver (v1) |
| Static vs dynamic linking | Fully static (`-static`) |
| Library discovery | CMake-configured paths baked into binary at build time |
| Default optimization | O0 (will raise later) |
| Binary name | `eco-boot` (no collision — JS artifacts end in `.js`) |

## Current State

| Capability | Status |
|---|---|
| MLIR parsing + lowering pipeline | Exists (`EcoPipeline.cpp`) |
| MLIR → LLVM IR translation | Exists (`dumpLLVMIR()` in `ecoc.cpp`) |
| JIT execution | Exists (`runJIT()` in `ecoc.cpp`, `EcoRunner`) |
| LLVM IR → object file (.o) | **Not implemented** |
| Linking to ELF binary | **Not implemented** |
| Frontend invocation (Node.js) | **Not implemented** (E2E tests do this ad-hoc) |

Available on system: LLVM 21.1.4, `lld`, `clang`, full x86 codegen libraries.

## Architecture

```
                     eco-boot
                    ┌─────────────────────────────────┐
 hello.elm ──────>  │  1. Invoke Node.js frontend      │
   (or .mlir)       │     node eco-boot-2-runner.js     │
                    │     make --output=tmp.mlir ...    │
                    │     (skipped if input is .mlir)   │
                    │                                   │
                    │  2. Parse .mlir (Eco dialect)     │
                    │                                   │
                    │  3. MLIR pipeline                 │
                    │     (Eco → LLVM dialect)          │
                    │                                   │
                    │  4. Translate → LLVM IR           │
                    │                                   │
                    │  5. LLVM opt passes (O0 default)  │
                    │                                   │
                    │  6. Emit object file (.o)         │
                    │     via TargetMachine::emit()     │
                    │                                   │
                    │  7. Link with clang++ driver:     │
                    │     - eco_entry.o (main wrapper)  │
                    │     - libEcoRuntime.a             │
                    │     - libElmKernel_*.a            │
                    │     - libEcoKernel_*.a            │
                    │     -static -lpthread -lm         │
                    │     → hello (ELF executable)      │
                    └─────────────────────────────────┘
```

## Detailed Design

### Step 1: Entry Point Wrapper (`eco_entry.cpp`)

A small C++ file compiled as a static library, providing the real `main()` for output binaries:

```cpp
// runtime/src/codegen/eco_entry.cpp
#include "allocator/Allocator.hpp"

extern "C" {
    // Generated by MLIR codegen (may not exist for simple modules):
    __attribute__((weak)) void __eco_init_globals();

    // The Elm main, renamed from "main" to avoid clash with C main:
    int64_t eco_main();

    // Effect manager registration (from EcoKernel):
    void eco_register_all_effect_managers();
}

int main(int argc, char** argv) {
    Elm::Allocator::instance().initialize();
    Elm::Allocator::instance().initThread();

    if (__eco_init_globals)
        __eco_init_globals();

    eco_register_all_effect_managers();

    int64_t result = eco_main();

    Elm::Allocator::instance().cleanupThread();
    return static_cast<int>(result);
}
```

The MLIR codegen currently emits a function called `main`. For AOT compilation this conflicts with the C `main`. After translating MLIR to LLVM IR, we rename it:

```cpp
llvmModule->getFunction("main")->setName("eco_main");
```

`__eco_init_globals` is declared as a weak symbol so modules without `eco.global` ops link correctly (the weak symbol resolves to null).

### Step 2: Static Runtime Libraries (CMake)

Currently, runtime/allocator/kernel `.cpp` files are compiled directly into `ecoc` and `EcoRunner`. For `eco-boot`, we need them as **static libraries** that output binaries link against.

New CMake targets in `runtime/src/codegen/CMakeLists.txt`:

```cmake
# Static library: Eco runtime (GC, allocator, string/list ops, platform)
add_library(EcoRuntimeStatic STATIC
    ../allocator/RuntimeExports.cpp
    ../allocator/Allocator.cpp
    ../allocator/NurserySpace.cpp
    ../allocator/OldGenSpace.cpp
    ../allocator/RootSet.cpp
    ../allocator/ThreadLocalHeap.cpp
    ../allocator/GCStats.cpp
    ../allocator/StringOps.cpp
    ../allocator/ListOps.cpp
    ../allocator/BytesOps.cpp
    ../allocator/ElmBytesRuntime.cpp
    ../platform/Scheduler.cpp
    ../platform/PlatformRuntime.cpp
)

# Static library: Entry point wrapper for output binaries
add_library(EcoEntryStatic STATIC
    eco_entry.cpp
)
```

The existing `ElmKernel` (21 per-module static libraries) and `EcoKernel` (7 per-module static libraries) are already built by CMake as static libraries — they're ready to use.

**Path configuration**: CMake bakes the build-time library paths into `eco-boot` via `target_compile_definitions`:

```cmake
target_compile_definitions(eco-boot PRIVATE
    ECO_RUNTIME_LIB="$<TARGET_FILE:EcoRuntimeStatic>"
    ECO_ENTRY_LIB="$<TARGET_FILE:EcoEntryStatic>"
    ECO_LIB_DIR="${CMAKE_BINARY_DIR}"
)
```

This means `eco-boot` knows exactly where the `.a` files are in the build tree — no runtime discovery needed. The paths are absolute and resolved at CMake configure/generate time.

### Step 3: Object File Emission (`emitObjectFile()`)

New function in `eco-boot.cpp`:

```cpp
static int emitObjectFile(llvm::Module &llvmModule,
                          llvm::TargetMachine &tm,
                          const std::string &outputPath) {
    std::error_code ec;
    llvm::raw_fd_ostream dest(outputPath, ec, llvm::sys::fs::OF_None);
    if (ec) {
        llvm::errs() << "Could not open output file: " << ec.message() << "\n";
        return 1;
    }

    llvm::legacy::PassManager emitPM;
    if (tm.addPassesToEmitFile(emitPM, dest, nullptr,
                                llvm::CodeGenFileType::ObjectFile)) {
        llvm::errs() << "Target machine can't emit object files\n";
        return 1;
    }
    emitPM.run(llvmModule);
    dest.flush();
    return 0;
}
```

**TargetMachine creation** uses the proper `TargetRegistry` API (not the JIT builder):

```cpp
static std::unique_ptr<llvm::TargetMachine> createTargetMachine(
    llvm::Module &module, unsigned optLevel) {
    auto triple = llvm::sys::getDefaultTargetTriple();
    module.setTargetTriple(triple);

    std::string error;
    auto *target = llvm::TargetRegistry::lookupTarget(triple, error);
    if (!target) { /* error */ }

    auto *tm = target->createTargetMachine(
        triple,
        "generic",     // CPU
        "",            // Features
        llvm::TargetOptions(),
        llvm::Reloc::PIC_,
        llvm::CodeModel::Small,
        static_cast<llvm::CodeGenOptLevel>(optLevel));

    module.setDataLayout(tm->createDataLayout());
    return std::unique_ptr<llvm::TargetMachine>(tm);
}
```

Required LLVM components (to add to CMake link list for eco-boot):
- `LLVMTarget`
- `LLVMX86CodeGen`
- `LLVMX86AsmParser`
- `LLVMX86Desc`
- `LLVMX86Info`
- `LLVMAsmPrinter`
- `LLVMCodeGen`
- `LLVMMC`
- `LLVMObject`
- `LLVMPasses` (for optimization pipeline)

### Step 4: Linking via `clang++` Driver

After emitting the `.o` file, invoke the system `clang++` as a linker driver via `llvm::sys::ExecuteAndWait`:

```cpp
static int linkExecutable(const std::string &objectFile,
                          const std::string &outputPath) {
    SmallVector<StringRef> args;
    args.push_back("clang++");
    args.push_back("-o");
    args.push_back(outputPath);
    args.push_back(objectFile);

    // Link the entry point wrapper
    args.push_back(ECO_ENTRY_LIB);    // baked in by CMake

    // Link the runtime
    args.push_back(ECO_RUNTIME_LIB);  // baked in by CMake

    // Link all ElmKernel module libraries
    // (paths collected from CMake at build time)
    for (const auto &lib : ecoElmKernelLibs)
        args.push_back(lib);

    // Link all EcoKernel module libraries
    for (const auto &lib : ecoKernelLibs)
        args.push_back(lib);

    // System libraries for fully static linking
    args.push_back("-static");
    args.push_back("-lpthread");
    args.push_back("-lm");
    args.push_back("-lstdc++");
    args.push_back("-lc");

    std::string errMsg;
    int rc = llvm::sys::ExecuteAndWait(
        "clang++", args, /*env=*/std::nullopt,
        /*redirects=*/{}, /*secondsToWait=*/0, /*memoryLimit=*/0, &errMsg);
    if (rc != 0) {
        llvm::errs() << "Linking failed: " << errMsg << "\n";
        return 1;
    }
    return 0;
}
```

`clang++` handles locating `crt1.o`, `crti.o`, `crtn.o`, `libc.a`, `libgcc.a`, etc. — all the platform-specific startup and C runtime files. This is why we use it as a driver rather than calling `lld` directly.

### Step 5: Frontend Invocation

`eco-boot` invokes the Node.js compiler frontend as a subprocess. Input type is detected by file extension:

```cpp
int compileToMlir(const std::string &inputFile,
                  const std::string &mlirOutput) {
    // If input is already .mlir, just copy/use it directly
    if (inputFile.ends_with(".mlir"))
        return 0;  // use inputFile as mlirOutput directly

    SmallVector<StringRef> args;
    args.push_back("node");
    args.push_back(frontendRunner);  // eco-boot-2-runner.js path
    args.push_back("make");
    args.push_back("--output=" + mlirOutput);
    // Pass through any --kernel-package / --local-package flags
    args.push_back(inputFile);

    int rc = llvm::sys::ExecuteAndWait("node", args, ...);
    return rc;
}
```

The runner path (eco-boot-2-runner.js) is configurable via `--frontend=<path>` CLI flag.

### Step 6: `eco-boot` CLI Design

```
eco-boot <input.elm|input.mlir> [options]

Options:
  -o <file>              Output path (default: a.out for exe, input.o for obj)
  --emit=<mode>          Output mode:
      exe (default)        Emit linked ELF executable
      obj                  Emit object file only (no linking)
      llvm                 Emit LLVM IR (text)
      mlir                 Emit MLIR only (frontend only, .elm input only)
  -O<n>                  Optimization level (0-3, default: 0)
  --frontend=<path>      Path to Node.js runner script (eco-boot-2-runner.js)
  --verbose              Print subcommands being executed
```

### Step 7: Effect Manager Handling

`eco_register_all_effect_managers()` is defined in the EcoKernel libraries. It's called from the entry wrapper, same as in JIT. For programs that don't use effects, it registers nothing and is harmless.

## Implementation Plan

### Phase 1: Static Library Refactoring
1. Create `EcoRuntimeStatic` CMake target (extract runtime sources into a shared static lib)
2. Create `EcoEntryStatic` CMake target with `eco_entry.cpp`
3. Verify existing `ElmKernel` and `EcoKernel` static libraries work for static linking
4. Collect all library paths via CMake generator expressions for baking into eco-boot

### Phase 2: Object File Emission
5. Create `eco-boot.cpp` based on `ecoc.cpp`, adding:
   - `createTargetMachine()` using `TargetRegistry::lookupTarget()` (not JIT builder)
   - `emitObjectFile()` using `TargetMachine::addPassesToEmitFile()`
   - `main` → `eco_main` rename in LLVM IR after translation
6. Add required LLVM codegen libraries to CMake link list
7. Test: `eco-boot input.mlir --emit=obj -o test.o` then verify with `objdump -t test.o`

### Phase 3: Linking
8. Implement `linkExecutable()` that invokes `clang++` with all static libraries and `-static`
9. Bake library paths into eco-boot via CMake `target_compile_definitions`
10. Test: `eco-boot input.mlir -o test && ./test` — verify output matches JIT

### Phase 4: Frontend Integration
11. Implement `compileToMlir()` subprocess invocation (detect `.elm` vs `.mlir` by extension)
12. Wire up CLI: `eco-boot hello.elm -o hello`
13. Handle temp file management (create temp `.mlir`, clean up after)

### Phase 5: E2E Verification
14. Add CMake target `eco-boot-check` that compiles each E2E test `.elm` to a binary and runs it
15. Compare outputs against `-- CHECK:` patterns (same verification as JIT test harness)
16. Verify all currently-passing E2E tests also pass when compiled to native
