# Typed Closures: Eliminate PAP Wrappers

## Overview

This plan eliminates the MLIR-visible `_pap_wrapper` functions generated by Elm codegen by making the closure path "typed end-to-end". Instead of boxing all captured values and arguments to `!eco.value`, we:

1. Store values in closures in their actual SSA types (`i64`, `f64`, `!eco.value`)
2. Generate lambda functions with typed ABIs (not all-`!eco.value`)
3. Use the runtime's 52-bit unboxed bitmap to track which slots are primitives

**Clarification**: This eliminates wrappers at the MLIR level. The LLVM lowering may still generate internal `__closure_wrapper_*` functions to adapt from the `void**` evaluator convention to typed function calls - this is an implementation detail invisible to MLIR.

---

## Current Architecture (Problem)

### 1. Lambda Functions (`compiler/src/Compiler/Generate/MLIR/Lambdas.elm`)

**`generateLambdaFunc`** (find with `grep -n "generateLambdaFunc" compiler/src/Compiler/Generate/MLIR/Lambdas.elm`) currently:
- Uses **all-`!eco.value` signature** for captures and params (grep: `Types.ecoValue`)
- Generates internal `eco.unbox` for primitives
- Boxes return value back to `!eco.value` via `boxToEcoValue`

```elm
-- Current: all params use !eco.value
captureArgPairs = List.map (\( name, _ ) -> ( "%" ++ name, Types.ecoValue )) lambda.captures
paramArgPairs = List.map (\( name, _ ) -> ( "%" ++ name, Types.ecoValue )) lambda.params
```

### 2. PAP Wrapper Generation (`compiler/src/Compiler/Generate/MLIR/Lambdas.elm`)

**`generatePapWrapper`** (find with `grep -n "generatePapWrapper"`) generates wrappers that:
- Accept all-`!eco.value` parameters
- Unbox to primitives for the target function call
- Box the result back to `!eco.value`

### 3. Closure Creation (`compiler/src/Compiler/Generate/MLIR/Expr.elm`)

**`generateVarGlobal`** (find with `grep -n "generateVarGlobal"`) queues `_pap_wrapper` when the target has unboxed params.

**`generateClosure`** (find with `grep -n "generateClosure"`) boxes ALL captures via `boxArgsWithMlirTypes`.

### 4. Closure Application (`compiler/src/Compiler/Generate/MLIR/Expr.elm`)

**`generateClosureApplication`** (find with `grep -n "generateClosureApplication"`) boxes ALL arguments via `boxArgsWithMlirTypes`.

### 5. LLVM Lowering (`runtime/src/codegen/Passes/EcoToLLVMClosures.cpp`)

- `PapCreateOpLowering` (find with `grep -n "PapCreateOpLowering"`): Inlines allocation, computes bitmap from operand types
- `PapExtendOpLowering` (find with `grep -n "PapExtendOpLowering"`): **Uses runtime helpers** (`eco_pap_extend` for partial, `eco_closure_call_saturated` for saturated)

### 6. Runtime (`runtime/src/allocator/RuntimeExports.cpp`)

- `eco_pap_extend` (find with `grep -n "eco_pap_extend"`): **Assumes all new args are boxed** (grep: `new_closure->unboxed = old_unboxed`)

---

## Proposed Architecture (Solution)

### Design Decision: Option A - Explicit Attribute

Add `unboxed_bitmap` attribute to `eco.papCreate` (and optionally `eco.papExtend`), making it the **source-of-truth**. Elm codegen computes and sets the bitmap; lowering trusts it (with optional verification).

This mirrors how `eco.construct.tuple2`, `eco.construct.record`, and `eco.construct.custom` use `unboxed_bitmap`.

#### Step 0.3: Update Verifier for `eco.papCreate`

**File**: `runtime/src/codegen/EcoOps.cpp`

Add verification that:
1. `unboxed_bitmap` only has bits set for indices < `num_captured`
2. `unboxed_bitmap` bits match operand types (non-`!eco.value` operand = bit set)
3. `unboxed_bitmap` fits in 52 bits (runtime Closure struct constraint)
4. `num_captured` Ōēż 63 (6-bit `n_values` field)
5. `arity` Ōēż 63 (6-bit `max_values` field)

```cpp
mlir::LogicalResult PapCreateOp::verify() {
    uint64_t bitmap = getUnboxedBitmap();
    int64_t numCaptured = getNumCaptured();
    int64_t arity = getArity();

    // Bitmap must fit in 52 bits
    if (bitmap >= (1ULL << 52)) {
        return emitOpError("unboxed_bitmap exceeds 52-bit capacity");
    }

    // No bits set beyond num_captured
    uint64_t validMask = (1ULL << numCaptured) - 1;
    if (bitmap & ~validMask) {
        return emitOpError("unboxed_bitmap has bits set beyond num_captured");
    }

    // Closure struct limits
    if (numCaptured > 63) {
        return emitOpError("num_captured exceeds 6-bit n_values limit (63)");
    }
    if (arity > 63) {
        return emitOpError("arity exceeds 6-bit max_values limit (63)");
    }

    // Verify bitmap matches operand types
    auto captured = getCaptured();
    for (size_t i = 0; i < captured.size(); ++i) {
        bool isBitSet = (bitmap >> i) & 1;
        bool isUnboxedType = !isa<ValueType>(captured[i].getType());
        if (isBitSet != isUnboxedType) {
            return emitOpError("unboxed_bitmap bit ") << i
                << " doesn't match operand type";
        }
    }
    return mlir::success();
}
```

#### Step 0.4: Add Verifier for `eco.papExtend`

**File**: `runtime/src/codegen/EcoOps.cpp`

mlir::LogicalResult PapExtendOp::verify() {
    uint64_t bitmap = getNewargsUnboxedBitmap();
    auto newargs = getNewargs();

    // Bitmap must fit in 52 bits
    if (bitmap >= (1ULL << 52)) {
        return emitOpError("newargs_unboxed_bitmap exceeds 52-bit capacity");
    }

    // No bits set beyond newargs size
    uint64_t validMask = newargs.empty() ? 0 : (1ULL << newargs.size()) - 1;
    if (bitmap & ~validMask) {
        return emitOpError("newargs_unboxed_bitmap has bits set beyond newargs count");
    }

    // Verify bitmap matches operand types
    for (size_t i = 0; i < newargs.size(); ++i) {
        bool isBitSet = (bitmap >> i) & 1;
        bool isUnboxedType = !isa<ValueType>(newargs[i].getType());
        if (isBitSet != isUnboxedType) {
            return emitOpError("newargs_unboxed_bitmap bit ") << i
                << " doesn't match operand type";
        }
    }
    return mlir::success();
}

-- Only i64 and f64 are unboxable; all other non-eco.value types

isUnboxableType : MlirType -> Bool

    mlirTy == Types.i64 || mlirTy == Types.f64

    List.indexedMap

## Testing Strategy

1. **Smoke test**: Simple `f 1` partial application where `f : Int -> Int -> Int`
2. **Mixed types**: Closures capturing `Int`, `Float`, `String` (boxed)
3. **Nested closures**: `\x -> \y -> x + y`
4. **Saturated calls**: Verify `eco_closure_call_saturated` works with typed args
5. **Edge cases**:
   - 0 captures
   - 52 captures (max for bitmap)
   - 53 captures (should fail compilation)
   - Arity 63 (max)
   - Arity 64 (should fail)

---

## Resolved Design Decisions

1. **Capture/Arity Limits**: Compile-time error at 52 captures or 63 arity
2. **Kernel Functions**: No "all-i64" fallback; fail if type unknown (consistent with UndefinedFunction pass)
3. **`eco.call`**: No changes; closure application routes through `eco.papExtend`
4. **Attribute vs computed bitmap**: Option A - `unboxed_bitmap` attribute is source-of-truth

---

## Assumptions

1. All unboxable primitives fit in 64 bits (i64, f64, i16/Char embedded)
2. Target function signatures available during LLVM lowering (UndefinedFunction pass)
3. GC correctly handles unboxed bitmap (already works for tuples/records/custom)
4. Evaluator convention (`void*(*)(void**)`) unchanged; wrappers handle typed calls
