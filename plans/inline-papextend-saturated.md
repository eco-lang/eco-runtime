# Inline PapExtend Saturated Calls

## Problem

`PapExtendOpLowering`'s saturated case (when `numNewArgs == remainingArity`) calls the
C runtime helper `eco_closure_call_saturated`. This introduces:

1. **Unnecessary function call overhead** — the helper does work (resolve closure, load fields,
   copy captured values, call evaluator) that can be emitted as inline LLVM ops.
2. **Redundant bitcasts** — f64 args must be bitcast to i64 for the helper's `uint64_t*` array,
   and the i64 result must be bitcast back to f64. With inline ops and opaque pointers, the
   arg-side bitcasts are unnecessary.
3. **Code duplication** — `CallOpLowering`'s indirect path already implements exactly the same
   inline logic. The two should share code.

## Current State

### PapExtendOpLowering saturated (lines 329–363)
```
1. Build uint64_t* args array, bitcast f64 args to i64 before storing
2. Call eco_closure_call_saturated(closure, args, numArgs) → i64
3. Bitcast i64 → f64 if needed
```

### CallOpLowering indirect (lines 422–521)
```
1. eco_resolve_hptr(closure) → closurePtr
2. Load packed field → extract n_values
3. Load evaluator function pointer
4. Alloca combined args array (n_values + numNewArgs)
5. Loop: copy captured values from closure struct to args array
6. Store new args directly (no f64 bitcast needed — opaque pointer store)
7. Indirect call through evaluator as ptr(ptr) → ptr
8. PtrToIntOp result → i64
```

### Key observations

- The CallOpLowering indirect path stores new args directly without bitcast (line 507:
  `StoreOp(newArgs[j], dstPtr)`). LLVM's opaque pointers allow storing any 64-bit type
  through a pointer. The evaluator wrapper handles the i64↔f64 conversion internally.
- The CallOpLowering indirect path has a **latent bug**: the result is always converted to
  i64 (line 517: `PtrToIntOp`), and the contBlock argument is typed i64 (line 470). If the
  `eco.call` result type were f64, this would be incorrect. Currently not triggered because
  indirect `eco.call` with f64 results doesn't occur in practice.
- The evaluator stored in closures is always a **wrapper** function with signature `ptr(ptr)`,
  generated by `getOrCreateWrapper` (lines 73–218). The wrapper loads i64 from the args array,
  bitcasts to the target type, calls the real function, and bitcasts the result back to ptr.
  This wrapper is unavoidable for dynamic dispatch.

## Plan

### Step 1: Extract shared helper function

Extract the inline closure-call logic from `CallOpLowering` into a static helper function:

```cpp
/// Emit inline LLVM ops to call a closure's evaluator with combined
/// (captured + new) arguments. Used by both papExtend-saturated and
/// indirect eco.call.
///
/// closureI64:  the closure HPointer as i64
/// newArgs:     the new arguments to append
/// resultType:  the expected LLVM result type (i64, f64, or ptr)
/// Returns the result Value with the correct type.
static Value emitInlineClosureCall(
    ConversionPatternRewriter &rewriter,
    Location loc,
    EcoRuntime &runtime,
    Value closureI64,
    ValueRange newArgs,
    Type resultType);
```

The logic is copied from `CallOpLowering` lines 441–518 with these changes:

- **Args storage**: Store `newArgs[j]` directly (no f64 bitcast — opaque pointer handles it).
- **Result conversion**: The evaluator wrapper always returns `ptr`. Converting to the final
  type requires:
  - `ptrtoint ptr → i64` (always required)
  - Branch to contBlock with i64 argument (contBlock always takes i64)
  - In contBlock: if `resultType == f64`, `bitcast i64 → f64`; if `resultType == ptr`,
    `inttoptr i64 → ptr`; if `resultType == i64`, no-op
- **No runtime sanity checks** — follows existing `CallOpLowering` pattern (trusts compiler).

**Rationale for result conversion**: Converting `ptr → f64` requires two ops (`ptrtoint` +
`bitcast`) — this is unavoidable with the current evaluator ABI (`ptr(ptr)`). Keeping contBlock
as `i64` simplifies the CFG (single block argument type) and puts the type-specific conversion
in one place.

### Step 2: Rewrite PapExtendOpLowering saturated case

Replace lines 329–363 with a call to the new helper:

```cpp
if (isSaturated) {
    Type convertedResultTy = getTypeConverter()->convertType(op.getResult().getType());
    Value result = emitInlineClosureCall(
        rewriter, loc, runtime, closureI64, newargs, convertedResultTy);
    rewriter.replaceOp(op, result);
}
```

This eliminates:
- The call to `eco_closure_call_saturated`
- The explicit f64→i64 bitcasts for args
- The explicit i64→f64 bitcast for result (handled inside the helper)

### Step 3: Rewrite CallOpLowering indirect case

Replace lines 422–521 with a call to the same helper:

```cpp
} else {
    // Indirect call through closure
    if (!op.getRemainingArity()) {
        return op.emitError("indirect calls require remaining_arity attribute");
    }
    int64_t remainingArity = op.getRemainingArity().value();
    auto allOperands = adaptor.getOperands();
    Value closureI64 = allOperands[0];
    auto newArgs = allOperands.drop_front(1);

    if (static_cast<int64_t>(newArgs.size()) != remainingArity) {
        return op.emitError("remaining_arity must equal number of new arguments");
    }

    Type convertedResultTy = resultTypes[0]; // already converted above
    Value result = emitInlineClosureCall(
        rewriter, loc, runtime, closureI64, newArgs, convertedResultTy);
    rewriter.replaceOp(op, result);
}
```

This also fixes the latent f64-result bug in the indirect call path.

#### Elm E2E tests (`test/elm/src/`)

Add **`IndirectCallFloatTest.elm`** — higher-order functions with Float returns.

**How to trigger indirect calls in Elm:**
- Pass a function as a parameter to another function
- Call that function parameter inside the body
- The compiler cannot statically determine which function it is → generates indirect call

**Test cases:**
```elm
-- Apply a Float->Float function (indirect call, f64 result)
applyFloat : (Float -> Float) -> Float -> Float
applyFloat f x = f x

-- Apply twice (chained indirect calls)
twiceFloat : (Float -> Float) -> Float -> Float
twiceFloat f x = f (f x)

-- Various float operations passed as closures
double x = x * 2.0
negate x = 0.0 - x
addTen x = x + 10.0

main =
    let
        _ = Debug.log "apply_double" (applyFloat double 5.0)     -- 10.0
        _ = Debug.log "apply_negate" (applyFloat negate 3.0)     -- -3.0
        _ = Debug.log "twice_double" (twiceFloat double 2.0)     -- 8.0
        _ = Debug.log "twice_addTen" (twiceFloat addTen 0.0)     -- 20.0
    in
    text "done"
```

**Why this triggers the bug path:**
- `applyFloat double 5.0` → compiler doesn't know `f` is `double` at the call site
- Generates `eco.call` with `remaining_arity` attribute (indirect call)
- Result type is f64 (Float) → exercises the `ptr → i64 → f64` conversion

## Decisions Made

| Issue | Decision | Rationale |
|-------|----------|-----------|
| contBlock argument type | Always `i64` | Simplifies CFG; type-specific conversion happens after in one place |
| Result conversion | `ptrtoint` + optional `bitcast`/`inttoptr` inside helper | Cannot avoid with current evaluator ABI (`ptr(ptr)`) |
| Sanity checks in inline code | None | Matches existing `CallOpLowering` pattern; trust compiler |
| Keep `eco_closure_call_saturated` | Yes, in runtime only | Used by `BytesExports.cpp`; remove only MLIR lowering helper |
| f64 indirect call test | Add 2 MLIR tests + 1 Elm E2E test | Exercises the fixed latent bug path at both MLIR and Elm levels |
