# Plan: Two-Pass Value Numbering for MLIR Bytecode Encoder

## Problem

The bytecode encoder in `IrSection.elm` assigns SSA value indices in a single pass
as it encodes ops. For non-isolated regions (like `eco.case` alternatives), MLIR's
bytecode format requires all alternatives to start value numbering from the SAME
base index. Our encoder accumulates indices across alternatives, causing mismatched
value references.

This produces 142 E2E test failures where the bytecode round-trip shows wrong
operand types, forward references to values defined after the use point, and
structural corruption.

## Root Cause

MLIR's value numbering algorithm (from `IRNumbering.cpp`):

```
For each op with regions:
    opFirstValueID = isIsolated ? 0 : nextValueID
    For each region in op.regions:
        push (region, opFirstValueID)   ← ALL regions get the SAME base

Worklist processing:
    pop (region, startID)
    nextValueID = startID               ← reset to base before each region
    number(region)                       ← numbers block args + op results sequentially
```

Our encoder does NOT reset `nextValueIndex` between alternatives. It threads the
value env through ops sequentially, so alternative 1 starts where alternative 0
ended. This shifts all value indices in later alternatives and beyond.

## Evidence

`AnonymousFunctionTest`, function `List_foldrHelper_$_12`:

- **Text format**: `eco.int.add(%ctr, %35)` — both operands defined before use ✓
- **Our bytecode round-trip**: `eco.int.add(%arg2, %22)` — `%22` is a `eco.papExtend`
  defined AFTER the `eco.case`, with type `!eco.value` instead of `i64`
- **Reference bytecode round-trip**: `eco.int.add(%arg2, %c1_i64)` — correct ✓

The reference bytecode (generated by MLIR from the same text input) has correct
value numbering because it resets the base index for each case alternative.

## Solution: Two-Pass Encoding

### Pass 1: Number all values in the isolated region

Walk the full IR tree within each isolated region (func.func body) and assign
sequential value indices to every block arg and op result. This produces a
complete `ValueEnv` (name → index map) before any encoding begins.

**Key rules** (matching MLIR's `IRNumberingState`):
- Block args: numbered sequentially in block order
- Op results: numbered sequentially after block args
- Non-isolated regions (eco.case alternatives): ALL start from the SAME base
  index (the `nextValueIndex` at the point the parent op's results were registered).
  After all alternatives, `nextValueIndex` advances to the MAX across alternatives.
- Isolated regions (func.func): numbering resets to 0, independent of parent

### Pass 2: Encode using pre-computed indices

Use the existing encoding functions but look up operand indices in the
pre-computed `ValueEnv` instead of building it incrementally.

## Implementation Steps

### Step 1: Add `numberRegion` function (~30 lines)

```elm
numberRegion : ValueEnv -> MlirRegion -> ValueEnv
```

Walks all blocks in a region, registering block args and op results.
Returns the env with all values registered and `nextValueIndex` advanced.

### Step 2: Add `numberOp` function (~25 lines)

```elm
numberOp : ValueEnv -> MlirOp -> ValueEnv
```

Registers the op's results, then processes its regions:
- For isolated ops (`func.func`): skip (regions numbered independently)
- For non-isolated ops (`eco.case`, `scf.if`, `scf.while`): number all
  alternative regions starting from the SAME base, then set `nextValueIndex`
  to the max across alternatives

### Step 3: Add `numberBlock` function (~10 lines)

```elm
numberBlock : ValueEnv -> MlirBlock -> ValueEnv
```

Registers block args, then calls `numberOp` for body ops + terminator.

### Step 4: Modify `encodeRegion` to use two-pass

Change `encodeRegion` to:
1. Call `numberRegion` on the input `ValueEnv` to get the complete env
2. Pass the complete env to `encodeBlock` / `encodeOp` for encoding

For isolated regions (called from `encodeOp` for `func.func`), pass
`emptyValueEnv` to `numberRegion` first.

### Step 5: Simplify `encodeOp` operand encoding

Currently `encodeOp` looks up operands in `valueEnv` (before this op's results).
With the two-pass approach, the env already has ALL values. Change operand
lookup to use the pre-computed env directly.

### Step 6: Remove incremental value env threading

The `encodeBlock` and `encodeOp` functions currently return `( BE.Encoder, ValueEnv )`
to thread the evolving value env. With the pre-computed env, they only need to
return `BE.Encoder`. Simplify the signatures.

### Step 7: Fix `countRegionValues` for non-isolated regions

`countRegionValues` currently counts all values across all alternatives independently.
For non-isolated regions (eco.case), it should count values using the SAME base
logic: each alternative starts from the same base, and the total is the max across
alternatives.

This is critical because the bytecode's `numValues` field in the region header tells
the reader how many values to pre-allocate. If this doesn't match the numbering,
the reader's value array will be the wrong size.

### Step 8: Validate with E2E tests

Run `cmake --build build --target full` (bytecode default) and compare against
the text-mode baseline (18 failures). Target: same 18 failures in bytecode mode.

## Detailed Design for `numberOp` (the critical function)

```elm
numberOp : ValueEnv -> MlirOp -> ValueEnv
numberOp env op =
    let
        -- Register this op's results
        envAfterResults =
            registerValues op.results env
    in
    if List.isEmpty op.regions then
        envAfterResults
    else if isIsolatedOp op.name then
        -- Isolated regions don't affect parent numbering
        envAfterResults
    else
        -- Non-isolated regions: all start from envAfterResults,
        -- advance to the max across all regions
        let
            baseIndex =
                envAfterResults.nextValueIndex

            maxIndex =
                List.foldl
                    (\region acc ->
                        let
                            envAfterRegion =
                                numberRegion envAfterResults region
                        in
                        max acc envAfterRegion.nextValueIndex
                    )
                    baseIndex
                    op.regions
        in
        { envAfterResults | nextValueIndex = maxIndex }
```

The key insight: `numberRegion envAfterResults region` gives each region the
same starting env (with the same `nextValueIndex`). We then take the MAX of
all the resulting `nextValueIndex` values. This matches MLIR's algorithm.

Note: the `valueMap` in `envAfterResults` accumulates entries from ALL
alternatives. Values with the same name in different alternatives will
overwrite each other. This is fine because:
1. Values in different alternatives have the SAME base index
2. Only one alternative executes at runtime
3. The bytecode reader uses the value INDEX (not name), and the index
   is correct because each alternative starts from the same base

## Questions and Open Issues

### Q1: Do eco.case alternatives define values with unique names?

If alternative 0 defines `%7` and alternative 1 also defines `%7` (same name),
the `valueMap` would store the index from whichever was processed last. Since
both alternatives start from the same base index, both `%7`s would get the
same index — which is correct for MLIR bytecode.

**But**: if alternative 0 defines `%7` at local offset 0 (= base+0) and
alternative 1 defines `%7` at local offset 2 (= base+2 because it defined
other values first), then `%7` gets index base+2 in the map, which is wrong
for alternative 0.

**Need to verify**: does the Elm AST use globally unique SSA names, or can
names collide across case alternatives? If names collide, we need separate
`valueMap` instances for each alternative (keeping only the base index
shared, not the name→index mapping).

### Q2: Does `countRegionValues` need the same fix?

`countRegionValues` currently sums values across all blocks in a region.
For non-isolated regions with multiple alternatives, should it count the
MAX across alternatives (since they share the value space) or the SUM?

MLIR's `number(Region)` stores `numValues = nextValueID - firstValueID`.
Since all alternatives start from the same base but may define different
numbers of values, the MAX gives the correct `numValues`.

Currently `countRegionValues` is only called on a single `MlirRegion`,
not on the alternatives of an op. The alternatives are separate regions
within the op. So `countRegionValues` for a single alternative just counts
that alternative's values, which is correct. The `numValues` in the region
header is per-region.

However, the OUTER region that contains the `eco.case` op needs to count
the eco.case result plus the MAX of its alternatives' values. This might
already work if `countRegionValues` counts the eco.case result (1) and the
alternatives' values are counted by the eco.case's own region headers.

**Need to verify**: how does `numValues` interact with non-isolated nested regions.

### Q3: Can we test incrementally?

Yes. After implementing the numbering pass, we can verify individual functions
by comparing the bytecode round-trip output against the text format. Use:
```bash
diff <(mlir-opt --allow-unregistered-dialect test_text.mlir) \
     <(mlir-opt --allow-unregistered-dialect test_binary.mlir)
```

### Assumptions

- A1: The Elm MLIR AST uses globally unique SSA names within each func.func body
  (the Pretty printer assigns unique `%0`, `%1`, ... names, so names within a
  func body should not collide across case alternatives)
- A2: Block arguments in eco.case regions (if any) also follow the same numbering
- A3: `scf.if` and `scf.while` regions follow the same non-isolated semantics
  as eco.case (they're not isolated from above)
