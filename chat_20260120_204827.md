# Plan 2 — Fix nested cases causing invalid SCF regions ("expects 0 or 1 blocks")

## Problem

MLIR structured control flow operations (`scf.if` and `scf.index_switch`) require each of their regions to have **exactly one block**. The failure `scf.if op expects region #0 to have 0 or 1 blocks` occurs when the EcoToLLVM pass processes an `eco.case` that is nested inside an SCF region.

### SCF Region Requirements

Both `scf.if` and `scf.index_switch` have the same structural requirement — each region must contain exactly one block. This is fundamental to SCF's design; it represents structured, high-level control flow that can be easily reasoned about and optimized:

```mlir
scf.if %cond {
  ^single_block:    // MUST be exactly one block
    ...
    scf.yield %result
}

scf.index_switch %idx
case 0 {
  ^single_block:    // MUST be exactly one block
    ...
    scf.yield %result
}
case 1 {
  ^single_block:    // MUST be exactly one block
    ...
    scf.yield %result
}
```

This means the problem is **not specific to `scf.if`** — any `eco.case` nested inside ANY SCF region (whether `scf.if` or `scf.index_switch`) will fail when `CaseOpLowering` tries to create multiple blocks.

### Root Cause Analysis

The issue arises from the interaction between two passes:

1. **EcoControlFlowToSCF pass**: Correctly transforms the OUTER `eco.case` to `scf.if`, cloning the inner `eco.case` into `scf.if`'s then region. The inner case is NOT transformed because it's not in terminal position (followed by `eco.constant`, not `eco.return` or `scf.yield`).

2. **EcoToLLVM pass**: Later, `CaseOpLowering` processes the inner `eco.case` which is now inside `scf.if`'s then region. The pattern creates NEW BLOCKS in the parent region:

```cpp
// EcoToLLVMControlFlow.cpp lines 573, 582-587
Block *currentBlock = op->getBlock();           // scf.if's then block
Region *parentRegion = currentBlock->getParent(); // scf.if's then region

Block *mergeBlock = rewriter.createBlock(parentRegion);  // PROBLEM: adds block to scf.if region!

for (size_t i = 0; i < alternatives.size(); ++i) {
    Block *caseBlock = rewriter.createBlock(parentRegion);  // More blocks added!
    ...
}
```

This violates `scf.if`'s single-block region requirement.

### Sequence of Events

For `CaseListTest.elm` with nested list pattern matching:

```
Initial IR:
  outer_case {
    alt0: { "empty"; eco.return }
    alt1: {
      inner_case { "one"; eco.return }, { "many"; eco.return }
      eco.constant Unit
      eco.return
    }
  }
  eco.return

After SCF pass:
  scf.if {
    then_region {
      block0 {           // single block
        inner_case {...} // still eco.case, not transformed
        eco.constant
        scf.yield
      }
    }
    else_region {
      block0 { "empty"; scf.yield }
    }
  }
  eco.return

During EcoToLLVM pass (CaseOpLowering on inner_case):
  scf.if {
    then_region {
      block0 { ... cf.switch ... }   // original block, modified
      block1 { ... }                  // NEW - case block for alt0
      block2 { ... }                  // NEW - case block for alt1
      block3 { ... }                  // NEW - merge block
    }
    ...
  }

  ERROR: scf.if region has 4 blocks, expected 0 or 1
```

### Same Problem with scf.index_switch Inside scf.index_switch

The problem would be identical if an `eco.case` lowered to `scf.index_switch` contained a nested `eco.case`:

```

The issue is that `CaseOpLowering` uses **CF-style lowering** (creating multiple basic blocks with explicit branches) inside **SCF-style regions** (which require single blocks with structured terminators). This is incompatible regardless of which SCF operation is the parent.

## Target behavior

- Either bail out and let SCF-to-CF convert the parent first, or handle specially

void EcoToLLVMPass::runOnOperation() {
  mlir::ModuleOp module = getOperation();
  MLIRContext *ctx = module.getContext();

  // Build typeConverter + patterns ONCE (keeps your type+structure coupling intact).
  auto typeConverter = buildYourExistingTypeConverter(*ctx);
  mlir::RewritePatternSet patterns(ctx);
  buildAllYourExistingPatterns(typeConverter, patterns /* + everything */);

  constexpr int kMaxRounds = 20;

  for (int round = 0; round < kMaxRounds; ++round) {
    mlir::ConversionTarget target(*ctx);

    // 1) SCF must not survive any round.
    target.addIllegalDialect<mlir::scf::SCFDialect>();

    // 2) Configure legality exactly as in your original single-pass conversion:
    //    - LLVM dialect legal, builtin legal, func legal, etc.
    //    - cf dialect legality rules (including your eco.value constraints)
    //    - eco dialect ops illegal except where explicitly legalized, etc.
    configureYourExistingLegality(target, typeConverter);

    // 3) Critical: allow eco.case to remain *for this round* if it is nested under SCF.
    //    Otherwise, it must be rewritten by CaseOpLowering.
    target.addDynamicallyLegalOp(/*eco::CaseOp or name-based*/ [&](mlir::Operation *op) {
      if (op->getName().getStringRef() != "eco.case")
        return false; // not applicable; your existing legality handles other ops

      if (op->getParentOfType<mlir::scf::IfOp>() ||
          op->getParentOfType<mlir::scf::IndexSwitchOp>()) {
        return true; // temporarily legal this round
      }
      return false; // must be rewritten
    });

    // Run one coupled conversion round.
    // Use applyFullConversion if your target defines full legality;
    // otherwise applyPartialConversion, BUT it must succeed.
    if (failed(applyPartialConversion(module, target, patterns))) {
      module.emitError() << "EcoToLLVM: conversion failed in round " << round;
      signalPassFailure();
      return;

    // Check termination: if SCF is gone and eco.case is gone, we are done.
    bool hasSCF = containsSCF(module);
    bool hasEcoCase = containsEcoCase(module);

    if (!hasSCF && !hasEcoCase)

    // If SCF is gone but eco.case remains, that means the “temporarily legal under SCF”
    // rule can no longer apply, so the NEXT round must eliminate remaining cases.
    // The next round will rebuild legality and re-drive conversion, so the moved cases
    // are revisited and rewritten.
  }

  module.emitError() << "EcoToLLVM: exceeded max conversion rounds; likely stuck legality/pattern issue";
  signalPassFailure();
}
```

This is exactly the missing capability you described: “NOT converted YET but MUST be converted EVENTUALLY”. The “eventually” is achieved by the **next round**, not by hoping the same driver revisits it.

If you remove the dummy, then the only way for the enclosing region to still be well-formed is: **the last operation must itself be a terminator**. That’s what “make `eco.case` a terminator” would mean.

## 1) What does it mean for `eco.case` to be a terminator?

In MLIR, a block/region is structurally valid only if its last op is a terminator. Your `mkCaseRegionFromDecider` currently treats only:

`eco.return`, `eco.jump`, `eco.crash`, `eco.unreachable`

as terminators【】.

So if the last op is `eco.case`, `isValidTerminator` is false, and you append a dummy value + `eco.return`【】.

If you instead “allow `eco.case` to be a terminator”, then conceptually:

- The *enclosing* region terminates at `eco.case`.
- Control transfers into exactly one alternative region.
- And **that alternative region must itself terminate the enclosing control flow** (typically by `eco.return` / `eco.jump` / crash).

This matches your comment: “The dummy return is unreachable at runtime (eco.case branches always exit via their own returns)”【】.

### Important: this changes the meaning of `eco.case`
It is no longer an “expression that produces a value and continues”.
It is a “dispatch terminator” (like a switch/branch) that does not fall through.

That’s compatible with Elm’s exhaustiveness (always matches), *and* with your current CPS-ish decision tree lowering, but it’s **not** compatible with using `eco.case` as a general expression in the middle of a block unless you introduce joinpoints/continuations everywhere.

## 2) Why SCF lowering must change if `eco.case` is a terminator

Your SCF pass theory explicitly assumes a shape like:

> `eco.case ...` followed by a terminator (`eco.return` or `scf.yield`) which the rewrite can “replace” after building `scf.if/scf.index_switch` results【】.

But if `eco.case` becomes the terminator, there *is no* “next terminator op” after it.

So the rewrite cannot be “replace the following terminator”; it must become:

- Replace the *terminator* `eco.case` with:
  1. an `scf.if`/`scf.index_switch` that computes results, and then
  2. the appropriate terminator (`scf.yield` if we’re inside an SCF region; otherwise `eco.return`) in the same block.

Concretely, the rewrite becomes “terminator-form”:

### Example: terminator `eco.case` -> `scf.if` + terminator

Before (terminator form):
```mlir
// last op in the region
eco.case %cond [1, 0] result_types [!eco.value] {case_kind="bool"} {
  ... 
  eco.return %a : !eco.value
}, {
  ...
  eco.return %b : !eco.value
}
```

After (inside an SCF region, so we must end with `scf.yield`):
```mlir
%r = scf.if %cond -> (!eco.value) {
  // clone alt1 body, replace eco.return with scf.yield
  scf.yield %a : !eco.value
} else {
  scf.yield %b : !eco.value
}
scf.yield %r : !eco.value
```

After (at function/eco region boundary where `eco.return` is the terminator):
```mlir
%r = scf.if %cond -> (!eco.value) {
  scf.yield %a : !eco.value
} else {
  scf.yield %b : !eco.value
}
eco.return %r : !eco.value
```

This is consistent with your existing SCF lowering idea of replacing `eco.return` with `scf.yield` inside the cloned alternative regions【】—you just “emit the final terminator yourself” rather than rewriting a following op.

## 3) How does this help the nested-case problem?

It eliminates the reason you had `%12` / dummy ops in the first place.

Right now, you append dummy ops when the last op is `eco.case`【】. Those dummy ops make the inner `eco.case` “not in terminal position” for the SCF pass, so it doesn’t convert it, which leaves it nested inside SCF regions and then later CF-style lowering tries to create blocks and explodes.

If nested `eco.case` is itself a terminator, then the inner case becomes genuinely *structurally terminal*, and the SCF pass can rewrite it *in place* using the terminator-form rewrite above.

## 4) One more constraint you must resolve

Your dialect spec currently says for `eco.case`:

> “Each region must terminate with eco.return or eco.jump.”【】

If you make “terminator case” legal, then you must decide whether an alternative region is allowed to terminate with a nested terminator-case (transitively-returning), i.e. end with `eco.case` rather than `eco.return`.

So either:

- **Option A (strict):** alternatives must still end in `eco.return/eco.jump` (so nested terminator-case would be illegal), or
- **Option B (transitive):** allow an alternative to end with `eco.case` *if* that case is itself a terminator-form case.

If your goal is “no dummy ops”, you almost certainly want Option B.
