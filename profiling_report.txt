  BLOCKING BUG: Stack Overflow in Monomorphization

  Stage 5 crashes immediately upon starting monomorphization with RangeError: Maximum call stack size exceeded. The
  crash occurs in:

  Compiler.Monomorphize.TypeSubst.resolveMonoVarsHelp
    → List.map → List.foldr → List.foldrHelper
    → (recursive call back to resolveMonoVarsHelp)

  Location: compiler/src/Compiler/Monomorphize/TypeSubst.elm:339-370

  The function has cycle detection for MVar cases (via a visiting set), but the recursion overflows even with 12 MB
  of stack (default is ~1 MB). At 15+ MB stack, V8 segfaults instead of reporting a clean error. This suggests
  either:

  1. Pathological type expansion: The substitution map chains variable lookups through deeply nested types (e.g. a →
  MFunction [b] c, b → MFunction [d] e, ...) creating thousands of nesting levels
  2. Non-cyclic but exponential expansion: Types like a → Tuple(b, b), b → Tuple(c, c) expand to 2^N size while only
  being N levels deep, but each level's List.map/foldr adds stack frames proportional to the list length at that
  level

  This crash is new — it doesn't occur in Stages 2-4 (JS output), only in Stage 5 (MLIR output), which exercises the
  monomorphization path.

  Key Takeaways

  1. The immediate blocker is the monomorphization stack overflow. Stage 5 cannot complete. This is likely a
  regression from the recent global optimization efficiency work — the MLIR code path creates type structures that
  overflow the JS call stack during resolveMonoVarsHelp.
  2. Compilation phase is dominated by serialization and GC, not by actual compilation logic. The .ecot binary
  encoding (type encoders, string encoding, bytes write) accounts for ~20% of time, and GC pressure from all the
  object allocation accounts for 34-40%.
  3. The first run (fresh compile) spends the entire 5 minutes compiling modules and never reaches
  monomorphization/MLIR generation. The second run (cached .ecot) reaches monomorphization almost instantly but
  crashes.
  4. _Utils_update (record updates) appears in the top 30, suggesting frequent record copying. This is a known Elm
  performance characteristic.
  5. No profile data for the actual MLIR generation or global optimization phase could be collected because
  monomorphization crashes before those phases are reached.
