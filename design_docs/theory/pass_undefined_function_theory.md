# UndefinedFunction Pass

## Overview

The UndefinedFunction pass validates that all functions referenced by `eco.call` operations are defined or declared in the module. This enforces invariant CGEN_011: no undefined function symbols may escape MLIR codegen.

In practice, this pass catches undefined **kernel functions** (e.g., `Elm_Kernel_Basics_add`), since user-defined functions are always generated with definitions. If a kernel function call is emitted but its declaration is missing, this pass will fail with an error.

**File**: `runtime/src/codegen/Passes/UndefinedFunction.cpp`

## Pseudocode

```
FUNCTION runOnOperation(module):
    // Step 1: Collect all defined/declared function names
    definedFunctions = SET()
    FOR EACH funcOp IN module:
        definedFunctions.add(funcOp.getName())

    // Step 2: Check all eco.call ops for undefined functions
    undefinedFunctions = []
    FOR EACH callOp IN module:
        callee = callOp.getCalleeAttr()
        IF callee IS NULL:
            CONTINUE  // Indirect call, skip

        calleeName = callee.getValue()
        IF calleeName NOT IN definedFunctions:
            undefinedFunctions.append({calleeName, callOp.getLoc()})

    // Step 3: Fail build if any undefined functions found
    IF NOT undefinedFunctions.empty():
        FOR EACH {name, loc} IN undefinedFunctions:
            emitError(loc) << "undefined function: " << name
        signalPassFailure()
```

## Purpose

This pass validates invariant CGEN_011: every function referenced by `eco.call` must have a corresponding `func.func` definition or declaration.

The MLIR codegen phase is responsible for:
1. Generating definitions for user-defined functions
2. Generating declarations for kernel functions (e.g., `Elm_Kernel_Basics_add`)

By the time this pass runs, all function declarations must already exist. This pass simply confirms that invariant holds, failing the build if any undefined function is found.

## Pre-conditions

1. Input module contains valid ECO dialect IR
2. MLIR codegen has completed, including generation of all kernel function declarations
3. All `eco.call` operations have valid callee attributes or are indirect calls

## Post-conditions

1. If pass succeeds: all functions referenced by `eco.call` have definitions/declarations
2. If pass fails: build terminates with error messages listing undefined functions

## Invariant Enforced

**CGEN_011**: Every function referenced by `eco.call` must have a corresponding `func.func` definition or declaration in the module so no undefined function symbols escape MLIR codegen.

## Example: Pass Success

```mlir
module {
    // Kernel declaration (generated by MLIR codegen)
    func.func private @Elm_Kernel_Basics_add(!eco.value, !eco.value) -> !eco.value

    func.func @user_defined(%arg: !eco.value) -> !eco.value {
        %result = eco.call @Elm_Kernel_Basics_add(%arg, %arg) : (!eco.value, !eco.value) -> !eco.value
        eco.return %result : !eco.value
    }
}
// Pass succeeds: @Elm_Kernel_Basics_add is declared
```

## Example: Pass Failure

```mlir
module {
    func.func @user_defined(%arg: !eco.value) -> !eco.value {
        // ERROR: @Elm_Kernel_Basics_add not declared
        %result = eco.call @Elm_Kernel_Basics_add(%arg, %arg) : (!eco.value, !eco.value) -> !eco.value
        eco.return %result : !eco.value
    }
}
// Pass fails with: error: undefined function: Elm_Kernel_Basics_add
```

## Handling Indirect Calls

Indirect calls (function pointers, closure calls) don't have a callee attribute:

```mlir
// Direct call - has @symbol, validated by this pass
eco.call @some_function(%arg) : (!eco.value) -> !eco.value

// Indirect call - no @symbol, first operand is function value, skipped
eco.call %fn_ptr(%arg) : (!eco.value, !eco.value) -> !eco.value
```

The pass skips indirect calls since the callee isn't known at compile time.

## Relationship to MLIR Codegen

The MLIR codegen phase (`Compiler.Generate.MLIRMono`) is responsible for:

1. **User functions**: Generated as `func.func` definitions with bodies
2. **Kernel functions**: Generated as `func.func private` declarations (no body)

**Important**: Stubs for **all** kernel functions are generated during the Elm MLIR codegen phase. The types for these stubs are derived during **monomorphization**, which determines the concrete types for each kernel function call site. This means every kernel function that is called will have a corresponding declaration with the correct monomorphized signature.

Example kernel functions:
- `Elm_Kernel_Basics_add`, `_sub`, `_mul`, etc.
- `Elm_Kernel_List_cons`, `_map`, `_filter`, etc.
- `Elm_Kernel_String_append`, `_length`, etc.

If this pass fails, it indicates a bug in MLIR codegen that failed to generate a required declaration.

## Relationship to Other Passes

- **Runs After**: MLIR codegen (which generates all declarations)
- **Runs Before**: `EcoToLLVM` (which requires complete function declarations)
- **Purpose**: Catch codegen bugs early with clear error messages

## Why Not Generate Stubs?

Previous versions of this pass generated stub declarations for undefined functions. This was removed because:

1. **Masks bugs**: Silently generating stubs hides codegen bugs
2. **Wrong signatures**: Inferring signatures from call sites may be incorrect
3. **Single responsibility**: Codegen should own declaration generation
4. **Fail fast**: Better to fail early with clear error than link-time failure
