## Introduction

This document describes LLVM's statepoint and stack map infrastructure for garbage collection integration. It serves as a reference for implementing precise GC in ECO's MLIR/LLVM lowering pipeline.

### Purpose

Precise garbage collectors need to:
1. Identify every live pointer at collection points (safepoints)
2. Determine which objects each pointer references
3. Update pointers after objects are relocated

LLVM's statepoint mechanism provides this capability by generating stack maps that record the location of all GC-managed pointers at specific instruction addresses.

### Applicability to ECO

ECO uses a two-generation garbage collector (nursery + old generation) with copying collection in the nursery. During minor GC, objects are evacuated and pointers must be updated. Stack maps enable the runtime to find and update all stack-resident pointers during collection.

## Stack Maps Overview

### What Are Stack Maps?

**Stack maps** are data structures that record the locations of live values at specific instruction addresses. They answer the question: "At this program counter, which values are live and where are they stored (registers, stack slots, constants)?"

Stack maps are generated by LLVM at **safepoints** — locations where the garbage collector is allowed to run and potentially relocate objects. As LLVM's documentation states:

> "A safepoint is a location at which pointers visible to the compiled code (i.e. currently in registers or on the stack) are allowed to change."

### Why Stack Maps Are Needed for Precise GC

Without stack maps, a runtime cannot distinguish pointers from integers on the stack. Conservative GC approaches scan the stack and treat anything that looks like a valid heap address as a potential pointer. This:
- Prevents object reclamation (false positives)
- Prohibits object relocation (cannot safely update "pointers" that might be integers)

Precise GC uses stack maps to know exactly:
- Which stack/register locations contain GC-managed pointers
- The base object each pointer references (for interior pointers)
- How to update those locations after relocation

### When Stack Maps Are Generated

Stack maps are generated at safepoints, which typically occur at:
- Function call sites (where control transfers to potentially long-running code)
- Loop back-edges (for long-running loops)
- Explicit safepoint polls (for cooperative preemption)

For ECO, we primarily need safepoints at allocation sites and function calls, since those are where GC can be triggered.

### Relationship to Garbage Collection

The integration flow is:

1. **Compile-time**: MLIR/LLVM lowering inserts `@llvm.experimental.gc.statepoint` intrinsics at safepoints
2. **Code generation**: LLVM backend generates stack map records in a special section (`.llvm_stackmaps`)
3. **Load-time**: Runtime parses stack map section and builds lookup structures
4. **Collection-time**: GC walks the stack, uses stack maps to find pointers, updates them after evacuation

## Statepoint Intrinsics

The statepoint mechanism uses three intrinsics that work together to enable explicit relocation in SSA form.

### Core Design Principle

Since SSA form prohibits modifying existing values, LLVM makes relocations explicit: after a statepoint where GC may occur, you must use `@llvm.experimental.gc.relocate` to get new SSA values representing potentially-updated pointers.

### The Three Intrinsics

#### 1. `@llvm.experimental.gc.statepoint`

The main intrinsic marking a safepoint location. It wraps a function call and generates a stack map recording all live GC pointers.

**Syntax**:
```llvm
token @llvm.experimental.gc.statepoint(
  i64 <id>,           ; Unique statepoint ID
  i32 <numPatchBytes>,; Reserved bytes for patching (usually 0)
  func <target>,      ; Function being called
  i64 <numCallArgs>,  ; Number of actual arguments
  i64 <flags>,        ; Control flags
  ... <call args>,    ; Actual call arguments
  i64 <numTransitionArgs>, ; Transition args count
  ... <transition args>,   ; Transition arguments
  i64 <numDeoptArgs>,      ; Deopt args count
  ... <deopt args>         ; Deoptimization arguments
) [ "gc-live"(<live pointers>) ]
```

**Returns**: A `token` value used by `gc.result` and `gc.relocate`

**Key parameters**:
- `id`: Arbitrary unique identifier for this statepoint
- `numPatchBytes`: Space reserved for runtime code patching (typically 0 for GC)
- `target`: The actual function being called (or a generic poll function for explicit safepoints)
- `flags`: Bitfield controlling behavior (e.g., whether GC can occur)
- `gc-live` bundle: Lists all GC pointers that must be tracked across this call

**Important**: The `gc-live` bundle is where you specify which pointers need relocation tracking. Only pointers listed here will appear in the generated stack map.

#### 2. `@llvm.experimental.gc.result`

Extracts the return value from the wrapped function call.

**Syntax**:
```llvm
type @llvm.experimental.gc.result(token %statepoint)
```

**Returns**: The original return value from the called function (type must match the called function's return type)

**Usage**: Call this immediately after the statepoint if you need the return value.

#### 3. `@llvm.experimental.gc.relocate`

Returns the potentially-updated value of a GC pointer after the safepoint.

**Syntax**:
```llvm
ptr @llvm.experimental.gc.relocate(
  token %statepoint,
  i32 <base_offset>,     ; Index of base pointer in gc-live bundle
  i32 <derived_offset>   ; Index of this pointer in gc-live bundle
)
```

**Returns**: New SSA value representing the potentially-relocated pointer

**Usage**: Must be called for every GC pointer you want to use after the statepoint. The `base_offset` and `derived_offset` are indices into the `gc-live` bundle:
- For simple pointers: both indices refer to the same pointer
- For derived pointers: `base_offset` points to the base object, `derived_offset` to the derived pointer

**Why base pointers?**: LLVM needs to track base pointers separately because:
- Interior pointers (e.g., `&obj->field`) need their base object to determine relocation
- Optimizations may compute derived pointers that survive longer than obvious uses of the base

### Statepoint Relocation Sequence

A complete statepoint sequence looks like:

```llvm
; Before statepoint: have original pointers
%obj1 = ...
%obj2 = ...

; Statepoint with gc-live bundle
%statepoint_token = call token (i64, i32, ptr, i32, i32, ...)
    @llvm.experimental.gc.statepoint.p0(
        i64 0,              ; ID
        i32 0,              ; Patch bytes
        ptr @some_function, ; Target
        i32 1,              ; Call arg count
        i32 0,              ; Flags
        i32 %arg,           ; Call argument
        i64 0, i64 0        ; No transition/deopt args
    ) [ "gc-live"(ptr %obj1, ptr %obj2) ]

; Get return value if needed
%result = call i32 @llvm.experimental.gc.result.i32(token %statepoint_token)

; Relocate all live GC pointers
%obj1.relocated = call ptr @llvm.experimental.gc.relocate(
    token %statepoint_token, i32 0, i32 0)  ; base=0, derived=0
%obj2.relocated = call ptr @llvm.experimental.gc.relocate(
    token %statepoint_token, i32 1, i32 1)  ; base=1, derived=1

; Use relocated pointers from here on
; (original %obj1, %obj2 are now invalid)
```

### Important Constraints

1. **Base pointers must remain live**: Even if the base pointer isn't used after the statepoint, LLVM requires it to remain in the `gc-live` bundle for correctness.

2. **All GC pointers must be relocated**: After a statepoint, you cannot use the original pointer SSA values. You must use values from `gc.relocate`.

3. **Architecture support**: Statepoints are fully supported only on x86_64 and AArch64.

4. **No write barriers**: The statepoint mechanism doesn't provide write barrier intrinsics. Those must be implemented with normal memory operations (relevant for generational collectors with old→young pointers, though ECO's immutability avoids this).

## Stack Map Binary Format

When LLVM generates code with statepoints, it emits a binary stack map section containing all safepoint information. The runtime must parse this at load time.

### Section Name

- **Darwin/macOS**: `__llvm_stackmaps` section in `__LLVM_STACKMAPS` segment
- **Linux/ELF**: `.llvm_stackmaps` section
- **Windows**: `.llvm_stackmaps` section

### Binary Structure

The stack map section follows this hierarchical layout (all values little-endian):

```c
struct StackMapSection {
    Header header;
    StkSizeRecord stkSizeRecords[header.numFunctions];
    uint64 constants[header.numConstants];
    StkMapRecord stkMapRecords[header.numRecords];
};
```

#### Header (24 bytes)

```c
struct Header {
    uint8  version;        // Always 3
    uint8  reserved1;
    uint16 reserved2;
    uint32 numFunctions;   // Count of StkSizeRecords
    uint32 numConstants;   // Count of large constants
    uint32 numRecords;     // Count of StkMapRecords
};
```

#### Function Stack Size Record (16 bytes)

One record per function containing statepoints:

```c
struct StkSizeRecord {
    uint64 functionAddress;  // Native code address of function
    uint64 stackSize;        // Frame size in bytes
    uint64 recordCount;      // Number of statepoints in this function
};
```

#### Stack Map Record (variable size)

One record per statepoint:

```c
struct StkMapRecord {
    uint64 patchpointID;     // The ID from gc.statepoint
    uint32 instructionOffset;// Offset from function start (bytes)
    uint16 reserved;
    uint16 numLocations;     // Count of location records

    Location locations[numLocations];  // Variable length

    uint32 padding;          // Align to 8 bytes
    uint16 numLiveOuts;      // Count of live registers
    LiveOut liveOuts[numLiveOuts];

    uint32 padding;          // Align to 8 bytes
};
```

#### Location Record (12 bytes)

Describes where a single value is stored:

```c
struct Location {
    uint8  locationType;     // Type enum (see below)
    uint8  reserved1;
    uint16 locationSize;     // Size in bytes
    uint16 dwarfRegNum;      // Register number (if applicable)
    uint16 reserved2;
    int32  offsetOrConstant; // Interpretation depends on type
};
```

**Location Types**:

| Type | Value | Meaning | offset/constant |
|------|-------|---------|-----------------|
| Register | 0x1 | Value in register | Unused |
| Direct | 0x2 | Frame-relative address | Offset from frame pointer |
| Indirect | 0x3 | Spilled to stack | Offset from register |
| Constant | 0x4 | Small immediate | The constant value |
| ConstIndex | 0x5 | Large constant | Index into constants array |

**Important**: "Direct" means the location record gives the *address* of the value, not the value itself. The value is at `[FramePointer + offset]`.

#### LiveOut Record (4 bytes)

Describes a register that's live out (contains a value used after the safepoint):

```c
struct LiveOut {
    uint16 dwarfRegNum;      // Register number
    uint8  reserved;
    uint8  sizeInBytes;      // Size of value in register
};
```

### Parsing Example (C++)

```cpp
struct StackMapParser {
    const uint8_t* data;
    size_t offset = 0;

    template<typename T>
    T read() {
        T value;
        std::memcpy(&value, data + offset, sizeof(T));
        offset += sizeof(T);
        return value;
    }

    void parse() {
        // Header
        uint8_t version = read<uint8_t>();
        assert(version == 3);
        offset += 3; // Skip reserved

        uint32_t numFunctions = read<uint32_t>();
        uint32_t numConstants = read<uint32_t>();
        uint32_t numRecords = read<uint32_t>();

        // Function records
        for (uint32_t i = 0; i < numFunctions; i++) {
            uint64_t functionAddr = read<uint64_t>();
            uint64_t stackSize = read<uint64_t>();
            uint64_t recordCount = read<uint64_t>();
            // Store function info...
        }

        // Constants
        std::vector<uint64_t> constants;
        for (uint32_t i = 0; i < numConstants; i++) {
            constants.push_back(read<uint64_t>());
        }

        // Stack map records
        for (uint32_t i = 0; i < numRecords; i++) {
            uint64_t patchpointID = read<uint64_t>();
            uint32_t instructionOffset = read<uint32_t>();
            offset += 2; // Skip reserved
            uint16_t numLocations = read<uint16_t>();

            std::vector<Location> locations;
            for (uint16_t j = 0; j < numLocations; j++) {
                Location loc;
                loc.type = read<uint8_t>();
                offset += 1; // Skip reserved
                loc.size = read<uint16_t>();
                loc.dwarfRegNum = read<uint16_t>();
                offset += 2; // Skip reserved
                loc.offsetOrConstant = read<int32_t>();
                locations.push_back(loc);
            }

            // Align to 8 bytes
            offset = (offset + 7) & ~7;

            uint16_t numLiveOuts = read<uint16_t>();
            for (uint16_t j = 0; j < numLiveOuts; j++) {
                offset += 4; // Skip liveout records
            }

            // Align to 8 bytes again
            offset = (offset + 7) & ~7;

            // Store record for this statepoint...
        }
    }
};
```

### Important Notes on Format

1. **Alignment**: Records are aligned to 8-byte boundaries. Padding bytes appear after location arrays and liveout arrays.

2. **Ordering preserved**: The order of locations matches the order of values in the statepoint's `gc-live` bundle.

3. **DWARF register numbering**: Register numbers use the platform's DWARF standard (e.g., x86_64 DWARF register numbers: rax=0, rdx=1, rcx=2, rbx=3, rsi=4, rdi=5, rbp=6, rsp=7, r8-r15=8-15).

4. **Size interpretation**: The `locationSize` field indicates the size of the pointer value, which for ECO's 40-bit logical pointers will need special handling (likely stored as 64-bit values).

## Integration Points

This section describes how ECO's MLIR/LLVM lowering and runtime will integrate with statepoints and stack maps.

### MLIR/LLVM Lowering Integration

#### Where to Insert Statepoints

The ECO MLIR lowering pass should insert `gc.statepoint` intrinsics at:

1. **Allocation sites**: Every call to the allocator (since allocation can trigger GC)
2. **Function calls**: Every call to user-defined Elm functions (may allocate internally)
3. **Explicit polls**: Loop back-edges for long-running computations (optional, for cooperative preemption)

#### Example MLIR → LLVM Lowering

Given Elm-like code:
```elm
let x = allocate(Cons, 16)
let y = allocate(Tuple2, 16)
let z = buildLargeList()
```

Lower to LLVM IR with statepoints:
```llvm
; Allocate first cons cell
%sp1 = call token (i64, i32, ptr, i32, i32, ...)
    @llvm.experimental.gc.statepoint.p0(
        i64 1, i32 0, ptr @eco_allocate, i32 2, i32 0,
        i32 16, i32 1,  ; size=16, tag=Cons
        i64 0, i64 0
    ) [ "gc-live"() ]  ; No live pointers yet
%x = call ptr @llvm.experimental.gc.result.p0(token %sp1)

; Allocate tuple (x is now live)
%sp2 = call token (i64, i32, ptr, i32, i32, ...)
    @llvm.experimental.gc.statepoint.p0(
        i64 2, i32 0, ptr @eco_allocate, i32 2, i32 0,
        i32 16, i32 2,  ; size=16, tag=Tuple2
        i64 0, i64 0
    ) [ "gc-live"(ptr %x) ]  ; x must survive
%y = call ptr @llvm.experimental.gc.result.p0(token %sp2)
%x.1 = call ptr @llvm.experimental.gc.relocate(token %sp2, i32 0, i32 0)

; Call function (x.1 and y are live)
%sp3 = call token (i64, i32, ptr, i32, i32, ...)
    @llvm.experimental.gc.statepoint.p0(
        i64 3, i32 0, ptr @buildLargeList, i32 0, i32 0,
        i64 0, i64 0
    ) [ "gc-live"(ptr %x.1, ptr %y) ]
%z = call ptr @llvm.experimental.gc.result.p0(token %sp3)
%x.2 = call ptr @llvm.experimental.gc.relocate(token %sp3, i32 0, i32 0)
%y.1 = call ptr @llvm.experimental.gc.relocate(token %sp3, i32 1, i32 1)
```

#### GC Strategy Selection

Mark ECO functions with the appropriate GC strategy:
```llvm
define ptr @myElmFunction() gc "statepoint-example" {
    ; ...
}
```

Or create a custom GC strategy plugin for ECO's specific needs.

#### Safepoint Insertion Passes

LLVM provides utility passes for automatic safepoint insertion:
- `PlaceSafepoints`: Inserts statepoint calls at function calls and loop back-edges
- `RewriteStatepointsForGC`: Rewrites IR to insert `gc.relocate` calls and maintain liveness

These can reduce manual work, but ECO may want fine-grained control over where safepoints occur.

### Runtime Integration

#### Loading Stack Maps

At program startup or JIT compilation time:

1. **Locate the section**: Use platform APIs to find `.llvm_stackmaps` section
2. **Parse into lookup structure**: Build a map from instruction address → stack map record
3. **Cache for fast lookup**: During GC, use instruction pointer to quickly find stack map

```cpp
class StackMapRegistry {
    std::unordered_map<uintptr_t, StackMapRecord> statepoint_map_;

public:
    void load_from_section(const uint8_t* data, size_t size) {
        StackMapParser parser{data};
        auto records = parser.parse();

        for (auto& record : records) {
            uintptr_t pc = record.function_addr + record.instruction_offset;
            statepoint_map_[pc] = std::move(record);
        }
    }

    const StackMapRecord* lookup(uintptr_t instruction_ptr) {
        auto it = statepoint_map_.find(instruction_ptr);
        return it != statepoint_map_.end() ? &it->second : nullptr;
    }
};
```

#### Stack Walking During GC

When GC is triggered:

1. **Capture thread state**: Get register context and stack bounds for each thread
2. **Walk the stack**: Iterate stack frames using frame pointers or platform unwinding
3. **For each return address**: Check if it's a statepoint location
4. **If statepoint found**: Use stack map to locate and process GC pointers

```cpp
void GarbageCollector::scan_stack_roots(ThreadContext& thread) {
    StackWalker walker(thread);

    while (walker.has_frames()) {
        uintptr_t return_addr = walker.return_address();

        // Check if return address is a statepoint
        auto* stack_map = stack_map_registry_.lookup(return_addr);
        if (!stack_map) {
            walker.next_frame();
            continue;
        }

        // Process each GC pointer location
        for (auto& location : stack_map->locations) {
            void** ptr_slot = resolve_location(location, walker.frame_pointer());

            if (ptr_slot && *ptr_slot) {
                Value* obj = logical_to_physical(*ptr_slot);
                Value* new_obj = evacuate_if_needed(obj);

                // Update pointer if object was moved
                if (new_obj != obj) {
                    *ptr_slot = physical_to_logical(new_obj);
                }
            }
        }

        walker.next_frame();
    }
}

void** resolve_location(const Location& loc, void* frame_ptr) {
    switch (loc.type) {
        case LocationType::Direct:
            // Address relative to frame pointer
            return (void**)((char*)frame_ptr + loc.offset);

        case LocationType::Indirect:
            // Spilled to stack at [reg + offset]
            void* base = get_register_value(loc.dwarf_reg_num);
            return (void**)((char*)base + loc.offset);

        case LocationType::Register:
            // Value in register (need to update saved context)
            return get_register_slot(loc.dwarf_reg_num);

        case LocationType::Constant:
        case LocationType::ConstIndex:
            // Constants don't need updating
            return nullptr;
    }
}
```

#### Root Set Registration

Unlike manually registered roots (from globals or explicit registration), stack roots are discovered dynamically during each GC cycle via stack walking. The runtime needs both:

- **Static roots**: Globals, thread-local storage → manually registered
- **Dynamic roots**: Stack and register roots → discovered via stack maps

#### Safepoint Polling

For cooperative preemption or allocation-free code with long-running loops, ECO may need safepoint polls:

```llvm
; Safepoint poll in loop
loop:
    ; Loop body...

    ; Poll for GC
    call token (i64, i32, ptr, i32, i32, ...)
        @llvm.experimental.gc.statepoint.p0(
            i64 99, i32 0, ptr @eco_safepoint_poll, i32 0, i32 0,
            i64 0, i64 0
        ) [ "gc-live"(ptr %live_ptr1, ptr %live_ptr2) ]
    %live_ptr1.1 = call ptr @llvm.experimental.gc.relocate(...)
    %live_ptr2.1 = call ptr @llvm.experimental.gc.relocate(...)

    br label %loop
```

The `eco_safepoint_poll` function can check a flag and trigger GC if needed.

### Thread Safety Considerations

- **Stack maps are read-only**: After parsing at load time, they don't change
- **Stack walking requires thread suspension**: All mutator threads must be stopped before scanning their stacks
- **Relocations are thread-local**: Each thread's stack pointers are updated independently

For ECO's architecture:
- Thread-local nurseries don't need synchronization for allocation
- Minor GC is thread-local: only that thread's stack needs walking
- Major GC is global: all threads must be suspended and all stacks walked

### Example 4: Parsing Stack Map at Runtime

**C++ Runtime Code**:
```cpp
#include <cstdint>
#include <vector>
#include <unordered_map>

struct Location {
    enum class Type : uint8_t {
        Register = 0x1,
        Direct = 0x2,
        Indirect = 0x3,
        Constant = 0x4,
        ConstIndex = 0x5
    };

    Type type;
    uint16_t size;
    uint16_t dwarf_reg;
    int32_t offset_or_const;
};

struct StackMapRecord {
    uint64_t patchpoint_id;
    uint32_t instruction_offset;
    std::vector<Location> locations;
};

class StackMapParser {
    const uint8_t* data_;
    size_t size_;
    size_t offset_ = 0;

    template<typename T>
    T read() {
        if (offset_ + sizeof(T) > size_) {
            throw std::runtime_error("Stack map parse error: unexpected end");
        }
        T value;
        std::memcpy(&value, data_ + offset_, sizeof(T));
        offset_ += sizeof(T);
        return value;
    }

    void align(size_t alignment) {
        offset_ = (offset_ + alignment - 1) & ~(alignment - 1);
    }

public:
    StackMapParser(const uint8_t* data, size_t size)
        : data_(data), size_(size) {}

    std::unordered_map<uintptr_t, StackMapRecord> parse() {
        // Header
        uint8_t version = read<uint8_t>();
        if (version != 3) {
            throw std::runtime_error("Unsupported stack map version");
        }
        offset_ += 3; // Reserved bytes

        uint32_t num_functions = read<uint32_t>();
        uint32_t num_constants = read<uint32_t>();
        uint32_t num_records = read<uint32_t>();

        // Function records
        std::vector<uint64_t> function_addrs;
        for (uint32_t i = 0; i < num_functions; i++) {
            uint64_t addr = read<uint64_t>();
            function_addrs.push_back(addr);
            offset_ += 16; // Skip stack size and record count
        }

        // Constants (not used for GC statepoints typically)
        offset_ += num_constants * sizeof(uint64_t);

        // Stack map records
        std::unordered_map<uintptr_t, StackMapRecord> result;

        for (uint32_t i = 0; i < num_records; i++) {
            StackMapRecord record;
            record.patchpoint_id = read<uint64_t>();
            record.instruction_offset = read<uint32_t>();
            offset_ += 2; // Reserved
            uint16_t num_locations = read<uint16_t>();

            // Parse locations
            for (uint16_t j = 0; j < num_locations; j++) {
                Location loc;
                loc.type = static_cast<Location::Type>(read<uint8_t>());
                offset_ += 1; // Reserved
                loc.size = read<uint16_t>();
                loc.dwarf_reg = read<uint16_t>();
                offset_ += 2; // Reserved
                loc.offset_or_const = read<int32_t>();
                record.locations.push_back(loc);
            }

            align(8);

            // LiveOuts (skip for now)
            uint16_t num_liveouts = read<uint16_t>();
            offset_ += num_liveouts * 4;

            align(8);

            // Map instruction address to record
            // Note: function_addr lookup would need function index from record
            // For simplicity, using instruction_offset as key
            uintptr_t pc = record.instruction_offset;
            result[pc] = std::move(record);
        }

        return result;
    }
};

// Usage
```

### Example 5: Stack Walking Pseudocode

**Pseudocode for GC Stack Scanning**:
```cpp
void scan_thread_stack(ThreadContext& thread) {
    void* sp = thread.stack_pointer;
    void* fp = thread.frame_pointer;
    void* stack_base = thread.stack_base;

    // Walk frames
    while (fp && fp < stack_base) {
        // Get return address from stack
        uintptr_t* frame = (uintptr_t*)fp;
        uintptr_t return_addr = frame[1];  // Platform-specific

        // Check if this frame has a statepoint
        StackMapRecord* record = lookup_stackmap(return_addr);
        if (record) {
            // Process each GC pointer location
            for (const Location& loc : record->locations) {
                void** ptr_location = nullptr;

                switch (loc.type) {
                    case Location::Type::Direct:
                        // Value on stack at fp + offset
                        ptr_location = (void**)((char*)fp + loc.offset_or_const);
                        break;

                    case Location::Type::Indirect:
                        // Spilled at [register + offset]
                        void* reg_val = get_saved_register(thread, loc.dwarf_reg);
                        ptr_location = (void**)((char*)reg_val + loc.offset_or_const);
                        break;

                    case Location::Type::Register:
                        // Value in saved register
                        ptr_location = get_saved_register_slot(thread, loc.dwarf_reg);
                        break;

                    case Location::Type::Constant:
                    case Location::Type::ConstIndex:
                        // Constants don't need updating
                        continue;
                }

                if (ptr_location && *ptr_location) {
                    // Process GC pointer
                    void* old_ptr = *ptr_location;
                    void* new_ptr = process_gc_root(old_ptr);

                    if (new_ptr != old_ptr) {
                        *ptr_location = new_ptr;
                    }
                }
            }
        }

        // Move to next frame
        fp = (void*)frame[0];  // Platform-specific
    }
}
```

## ECO-Specific Considerations

### 40-bit Logical Pointers

ECO uses 40-bit logical pointers (offsets into a unified heap) rather than native 64-bit addresses. This has implications for stack map integration:

#### Storage in Stack Maps

- LLVM will treat ECO pointers as 64-bit values (since the machine architecture is 64-bit)
- Stack map location records will specify 8-byte sizes
- The upper 24 bits should be zero for valid ECO pointers

#### Conversion During GC

When scanning stack roots:

```cpp
void* scan_eco_pointer(void** stack_slot) {
    uint64_t logical_ptr = (uint64_t)*stack_slot;

    // Validate it's an ECO pointer (top 24 bits should be zero or tag bits)
    if ((logical_ptr & 0xFFFFFF0000000000ULL) != 0) {
        // Not an ECO pointer (could be unboxed int, etc.)
        return *stack_slot;
    }

    // Convert to physical address
    Value* physical_obj = heap_base + (logical_ptr & 0xFFFFFFFFFFULL);

    // Process (evacuate if in nursery, mark if in old gen)
    Value* new_physical = process_object(physical_obj);

    // Convert back to logical pointer
    uint64_t new_logical = new_physical - heap_base;
    *stack_slot = (void*)new_logical;

    return (void*)new_logical;
}
```

#### Embedded Constants (Nil, True, False)

ECO embeds constants directly in pointer values. These must be distinguished from heap pointers in stack maps:

```cpp
bool is_heap_pointer(uint64_t logical_ptr) {
    // Check for embedded constants (e.g., special tag bits)
    if (logical_ptr == NIL_VALUE ||
        logical_ptr == TRUE_VALUE ||
        logical_ptr == FALSE_VALUE) {
        return false;
    }

    // Check for unboxed integers (if using tag bits)
    if ((logical_ptr & INT_TAG_MASK) == INT_TAG) {
        return false;
    }

    return true;
}
```

### Integration with NurserySpace and OldGenSpace

#### Minor GC (Nursery Collection)

When a thread triggers minor GC:

1. **Suspend the thread** (or run GC on the allocating thread)
2. **Walk that thread's stack** using stack maps
3. **For each stack root**:
   - Check if pointer points into nursery
   - If yes, evacuate object to to-space or promote to old gen
   - Update stack slot with new pointer

```cpp
void NurserySpace::collect_with_stackmaps() {
    ThreadContext& thread = current_thread();

    // Scan stack roots
    scan_stack_roots(thread, [this](Value** root_slot) {
        Value* obj = *root_slot;

        if (is_in_nursery(obj)) {
            Value* new_obj = evacuate(obj);
            *root_slot = new_obj;
        }
    });

    // Scan evacuated objects (Cheney's algorithm)
    complete_evacuation();
}
```

#### Major GC (Old Generation Collection)

When old generation is full:

1. **Stop all threads** (STW collection)
2. **Walk all threads' stacks** using stack maps
3. **Mark phase**: Mark all reachable objects
4. **Sweep phase**: Reclaim unmarked objects (old gen uses mark-sweep, not copying)

Since old gen doesn't move objects in the current design, stack pointers don't need updating (only marking).

```cpp
void OldGenSpace::collect_with_stackmaps() {
    // Stop the world
    suspend_all_threads();

    // Mark phase: scan all threads' stacks
    for (ThreadContext& thread : all_threads) {
        scan_stack_roots(thread, [this](Value** root_slot) {
            Value* obj = *root_slot;

            if (is_in_old_gen(obj)) {
                mark(obj);
            }
        });
    }

    // Sweep phase
    sweep_unmarked_objects();

    resume_all_threads();
}
```

### Thread-Local vs Global Considerations

- **Minor GC**: Thread-local, only scans the allocating thread's stack
- **Major GC**: Global, must scan all threads' stacks

This asymmetry is important for performance:
- Minor collections are frequent but cheap (single-threaded)
- Major collections are rare but expensive (multi-threaded coordination)

Stack map lookups must be fast since they happen on every GC cycle. Consider caching strategies:

```cpp
class StackMapCache {
    // Thread-local cache of recent lookups
    thread_local static std::array<CacheEntry, 8> cache_;

    static StackMapRecord* lookup(uintptr_t pc) {
        // Check cache first (fast path)
        for (auto& entry : cache_) {
            if (entry.pc == pc) {
                return entry.record;
            }
        }

        // Miss: lookup in global registry
        auto* record = global_registry.lookup(pc);

        // Update cache (LRU)
        std::rotate(cache_.begin(), cache_.begin() + 1, cache_.end());
        cache_[0] = {pc, record};

        return record;
    }
};
```

## Open Questions

### 1. LLVM Version Target

**Question**: Which LLVM version should ECO target?

**Considerations**:
- Statepoint mechanism has evolved across versions
- Newer versions have bug fixes and optimizations
- Need to balance cutting-edge features vs. stability

**Recommendation**: Target LLVM 17 or later (latest LTS as of 2024). Statepoints are considered stable by LLVM 15+.

### 2. Statepoint vs. GC.root

**Question**: Should ECO use statepoints or the older `llvm.gcroot` intrinsic?

**Analysis**:
- **gcroot**: Older, simpler, shadow-stack based
  - Pros: Less complex IR, works on all platforms
  - Cons: Performance overhead (separate shadow stack), deprecated

- **statepoint**: Modern, efficient, stack-map based
  - Pros: No shadow stack overhead, better optimizations, actively maintained
  - Cons: More complex IR, limited platform support (x86_64, AArch64)

**Recommendation**: Use statepoints. ECO targets x86_64 primarily, and the performance benefits outweigh the complexity. The `gcroot` approach is deprecated and may be removed from future LLVM versions.

### 3. Logical Pointer Representation

**Question**: How should 40-bit logical pointers be represented in LLVM IR?

**Options**:
1. Use native `i64` type, treat as opaque integers
2. Use LLVM's `ptr` type with address space attributes
3. Use custom non-integral pointer type

**Analysis**:
- **Option 1 (i64)**: Simple but loses pointer semantics, prevents LLVM optimizations
- **Option 2 (ptr with addrspace)**: Preserves pointer semantics, allows aliasing analysis
- **Option 3 (non-integral ptr)**: Most semantically correct, but complex integration

**Current thinking**: Option 2 seems best. Use `ptr addrspace(1)` for ECO heap pointers:

```llvm
; ECO pointer type
%obj = alloca ptr addrspace(1)

; Conversions
%logical = ptrtoint ptr addrspace(1) %obj to i64
%ptr_obj = inttoptr i64 %logical to ptr addrspace(1)
```

This allows LLVM to understand that they're pointers (for GC purposes) while distinguishing them from native pointers.

**Needs investigation**: How statepoints interact with non-default address spaces. Does `gc-live` bundle accept `ptr addrspace(1)`?

### 4. Stack Map Size and Performance

**Question**: What's the overhead of stack maps in terms of binary size and GC performance?

**Concerns**:
- Stack map section can be large (proportional to number of statepoints × live pointers)
- Parsing at load time adds startup cost
- Lookup during GC must be fast (potentially every frame of every thread)

**Needs measurement**:
- Build a prototype with statepoints
- Measure `.llvm_stackmaps` section size
- Benchmark GC pause times with stack walking
- Compare to shadow-stack approach

**Mitigation strategies**:
- Cache frequent lookups
- Use perfect hashing for stack map lookup
- Compress stack map section (if needed)

### 5. Safepoint Placement Strategy

**Question**: Where should safepoints be placed for optimal GC latency vs. throughput?

**Options**:
1. **Minimal**: Only at allocations (lowest overhead)
2. **Function calls**: At allocations + all function calls (moderate overhead)
3. **Loop back-edges**: Also in loops (highest overhead, best latency)

**Tradeoffs**:
- More safepoints → shorter max GC latency (threads reach safepoint faster)
- Fewer safepoints → better throughput (less relocation overhead)

**Elm-specific**: Elm code is typically allocation-heavy. Most computation involves building/transforming data structures. This suggests allocations + function calls may be sufficient.

**Recommendation**: Start with allocations + function calls. Add loop safepoints only if empirical testing shows unacceptable GC latency in long-running pure computations.

### 6. Inlining and Statepoints

**Question**: How do statepoints interact with function inlining?

**Concern**: If a function with statepoints is inlined, the stack map must reflect the inlined context. Does LLVM handle this correctly?

**Needs verification**:
- Create test case with inlined allocation
- Verify stack maps are correct after inlining
- Check if `gc-live` bundles are properly merged

**LLVM documentation says**: Statepoints are designed to work with inlining and other optimizations. The IR form should be preserved through optimization passes.

### 7. Exception Handling

**Question**: How do statepoints interact with exception handling (unwinding)?

**Concern**: If an exception is thrown at a statepoint, the runtime must:
1. Update GC pointers on the stack during unwinding
2. Ensure stack maps remain valid through cleanup code

**Elm-specific**: Elm doesn't have exceptions in user code, but the runtime might use C++ exceptions for errors.

**Needs investigation**:
- LLVM documentation mentions "exceptional edge relocations remain problematic"
- May need to avoid exceptions in GC-managed code
- Use explicit error returns instead

### 8. Concurrency and Stack Maps

**Question**: How do stack maps work with concurrent/parallel GC?

**Scenario**: If ECO moves to concurrent or parallel GC in the future:
- Multiple threads may be marking/evacuating simultaneously
- Stack maps from running threads must be read safely

**Considerations**:
- Statepoints assume stop-the-world for pointer updates
- Concurrent marking might work (read-only access to pointers)
- Concurrent evacuation would need careful synchronization

**For now**: ECO's design is stop-the-world, so not an immediate concern. But worth keeping in mind for future work.
