# MLIR Codegen Expectations from GlobalOpt and Monomorphization

## Overview
The MLIR code generation phase in Eco expects very specific preconditions from earlier phases (GlobalOpt and Monomorphization). This document details what the MLIR codegen assumes about MonoGraph data, MonoExpr structures, MonoType properties, and CallInfo metadata.

## Critical Entry Points

### Main Backend Entry: `Backend.elm`
- **Line 49-56**: `generateMlirModule` unpacks the MonoGraph
  - Expects: `MonoGraph { nodes, main, registry, ctorShapes }`
  - **CRITICAL**: Does NOT run monomorphization optimizer (commented out lines 53-54)
  - Uses `ctorShapes` directly - expects these to be complete and correct

### Main Processing: `Functions.elm`
- **Line 74**: `generateNode` processes each MonoNode by type
- **Line 71**: Uses `Functions.generateNode` for each node in the nodes dictionary

## Core Assumptions: MonoNode Structure

### 1. **Closure Invariant (MONO_004, GOPT_001)**
**Location**: `Functions.elm:82-133`, `Context.elm:564-596`

Every MonoNode with a function type MUST be one of:
- `MonoDefine expr monoType` where `expr` is `MonoClosure closureInfo body monoType`
- `MonoTailFunc params expr monoType` (tail-recursive variant)

**Failure Mode**: If a function-typed node is something else, MLIR codegen will crash when trying to extract closure parameters.

**MLIR Codegen Assumption**: 
- Function parameters are extracted from `closureInfo.params`
- Body is extracted from `closureInfo.body`
- If these don't exist, crash at line 156-172 in Functions.elm (pattern matching on MonoClosure)

---

### 2. **Closure Parameter Count vs Arity (GOPT_001)**
**Location**: `Expr.elm:720`, `Functions.elm:226, 288`

For every MonoClosure:
```elm
closureInfo.params : List (Name, MonoType)
```

**CRITICAL INVARIANT**: 
- `List.length closureInfo.params` must equal `stageParamTypes monoType`
- This is verified by TestLogic.Generate.MonoFunctionArity
- GlobalOpt's `canonicalizeClosureStaging` enforces this

**MLIR Impact**:
- Lines 206-208 (Functions.elm) build `argPairs` directly from `closureInfo.params`
- If param count doesn't match arity, function signature will be wrong
- Results in mismatched calling convention

---

### 3. **CtorLayout Completeness (CGEN_014)**
**Location**: `Backend.elm:64`, `Functions.elm:88-96`

The MonoGraph must arrive with `ctorShapes` already populated:
```elm
ctorShapes : EveryDict.Dict (List String) (List String) (List Mono.CtorShape)
```

**Monomorphization builds this**, GlobalOpt must preserve it.

**MLIR Codegen Usage**:
- Line 90-91 (Functions.elm): `Types.computeCtorLayout ctorShape`
- No layout reconstruction happens in MLIR
- If ctorShape data is malformed, `computeCtorLayout` will produce wrong field counts/bitmaps

## Core Assumptions: MonoExpr Structures

### 1. **ClosureInfo Captures (REP_CLOSURE_001, CGEN_003)**
**Location**: `Expr.elm:682-820`

ClosureInfo structure:
```elm
type alias ClosureInfo =
    { lambdaId : LambdaId
    , captures : List (Name, MonoExpr, Bool)
    , params : List (Name, MonoType)
    }
```

**What MLIR Codegen Expects**:

1. **Capture Expressions Evaluate Cleanly**
   - Lines 686-700: Each capture is evaluated with `generateExpr`
   - Result type is captured as actual SSA MlirType
   - ASSUMES: Capture expressions don't have side effects that vary between calls

2. **Captures Use Typed Closure ABI (CGEN_050)**
   - Line 702-709: Captures evaluated with actual SSA types
   - NO boxing/unboxing at closure creation
   - Line 732-739: Unboxed bitmap computed from actual MLIR types
   - **CRITICAL**: Only i64, f64, i16 are marked unboxed (not i1/Bool)

3. **Bool Must Not Be Unboxed (REP_CLOSURE_001, FORBID_CLOSURE_001)**
   - Line 727-739: `Types.isUnboxable mlirTy` check
   - `isUnboxable` only returns true for i64, f64, i16
   - Bool (i1) always marked as boxed (!eco.value)
   - If GlobalOpt produces captures with Bool marked unboxed, MLIR will generate wrong bitmap

---

### 2. **MonoCall Structure & CallInfo (CGEN_002, CGEN_052)**
**Location**: `Expr.elm:949-971, 998-1118`

```elm
type CallModel = FlattenedExternal | StageCurried

type alias CallInfo =
    { callModel : CallModel
    , stageArities : List Int
    , isSingleStageSaturated : Bool
    , initialRemaining : Int          -- CRITICAL
    , remainingStageArities : List Int -- CRITICAL
    , closureKind : MaybeClosureKind  -- ADDED: for typed closure calls
    , captureAbi : Maybe CaptureABI   -- ADDED: for typed closure calls
    , callKind : CallKind             -- ADDED: CallDirectKnownSegmentation, CallDirectFlat, CallGenericApply
    }
**UPDATED 2026-03-25**: CallInfo now has three additional fields: closureKind, captureAbi, callKind.
```

**Generated by**: GlobalOpt's annotateCallStaging pass (or defaultCallInfo if missing)

**MLIR Codegen Usage Pattern**:

1. **Dispatch Based on CallModel** (Line 951-970)
   - `FlattenedExternal`: ABI-flattened externals/kernels
   - `StageCurried`: User-defined closures with staging

2. **Saturated Call Decision** (Line 964-970)
   - Uses `isSingleStageSaturated` to pick code path
   - **CRITICAL**: If true, calls `generateSaturatedCall`
   - If false, calls `generateClosureApplication`

3. **Apply-by-Stages: initialRemaining** (Line 1276-1277)
   - `initialRemaining`: Stage arity of the closure at this call site
   - Used as `sourceRemaining` parameter to `applyByStages`
   - Line 1008-1118: Core staging loop uses this
   - **CRITICAL INVARIANT (CGEN_052)**: `remainingArity` in papExtend must equal `sourceRemaining` before application
   - Line 1056-1057: Enforced with comment "CGEN_052: remaining_arity is the SOURCE PAP's remaining"

4. **Remaining Stage Arities** (Line 1279-1280)
   - `remainingStageArities`: Arity sequence for stages AFTER this one
   - Used to determine what closure type is produced
   - Line 1065-1080: When current stage saturates, switches to next stage arity
   - If empty and stage saturates, result is final value (not closure)

## Core Assumptions: MonoType Properties

### 1. **Function Type Decomposition**
**Location**: `Types.elm:158-177`, `Expr.elm:1330-1626`

For every `Mono.MFunction` encountered:

1. **monoTypeToAbi Conversion (REP_ABI_001, CGEN_012)**
   ```elm
   monoTypeToAbi : MonoType -> MlirType
   ```
   - `MInt` → i64
   - `MFloat` → f64
   - `MChar` → i16
   - `MBool` → !eco.value (NOT i1)
   - Everything else → !eco.value

2. **monoTypeToOperand for Control Flow (REP_SSA_001)**
   - Used only for case scrutinee/if conditions
   - `MBool` → i1 (ONLY here is i1 valid)
   - All others same as ABI

3. **Decomposition via decomposeFunctionType** (Line 312, 609-610)
   - Recursively peels `MFunction [args] ret` layers
   - Returns `(List MonoType, MonoType)` = (all args, final return)
   - **ASSUMES**: Types are well-formed curried structures

---

### 2. **FuncSignature Extraction**
**Location**: `Context.elm:561-596`

For each MonoNode, `extractNodeSignature` determines what signature to register:

```elm
extractNodeSignature : Mono.MonoNode -> Maybe FuncSignature
```

**For MonoClosure**:
- Line 566-570: Extracts `List.map Tuple.second closureInfo.params` as paramTypes
- Return type: `Mono.typeOf body`

**For MonoTailFunc**:
- Line 578-591: Returns final return type after peeling MFunction
- **CRITICAL**: Uses pattern match on monoType

**For MonoCtor**:

## CallInfo Metadata Flow

### Where CallInfo is Created
**GlobalOpt Phase** (not shown, but effects MLIR):
- `Compiler.GlobalOpt.MonoInlineSimplify` or similar
- Runs `annotateCallStaging` to fill CallInfo fields
- Sets `callModel`, `stageArities`, `isSingleStageSaturated`, `initialRemaining`, `remainingStageArities`

### How MLIR Uses It
**Expr.elm generateCall**:

1. **Line 951**: Switch on `callInfo.callModel`
   - `FlattenedExternal`: Use ABI-flattened path
   - `StageCurried`: Check `isSingleStageSaturated` next

2. **Line 964-970**: For StageCurried:
   - If `isSingleStageSaturated`: All args provided in one stage → direct call
   - Otherwise: Multi-stage or partial → create closures

3. **Line 1276-1286**: Core closure application:
   - `initialRemaining = callInfo.initialRemaining`
   - `remainingStageArities = callInfo.remainingStageArities`
   - Call `applyByStages` with these values
   - Generate chain of `papExtend` ops

4. **Line 1095-1110**: papExtend attributes:
   ```elm
   papExtendAttrs =
       Dict.fromList
           [ ("_operand_types", ArrayAttr ...)
           , ("remaining_arity", IntAttr Nothing remainingArity)  -- MUST match sourceRemaining
           , ("newargs_unboxed_bitmap", IntAttr Nothing newargsUnboxedBitmap)
           ]
   ```
   - **remaining_arity MUST match the source PAP's remaining arity** (line 1056-1057, CGEN_052)
   - If GlobalOpt computes wrong initialRemaining, every papExtend will be wrong

## Boxing and Unboxing Assumptions

### Closure Boundary Normalization (CGEN_003)
**Location**: `Expr.elm:1296-1320`

At closure call sites, all arguments must be boxed:
- i1 (Bool) → !eco.value (via `boxToEcoValue`)
- i64, f64, i16 → unchanged (unboxable primitives)
- !eco.value → unchanged

```elm
boxArgsForClosureBoundary : Ctx.Context -> List (String, MlirType) -> ...
```

**ASSUMES**: Values have correct SSA types from code generation.

### Signature-Based Boxing (CGEN_007)
**Location**: `Expr.elm:863-908`

Before calling a function:
```elm
boxToMatchSignatureTyped : Ctx.Context 
    -> List (String, MlirType)
    -> List Mono.MonoType
    -> ...
```

Compares actual SSA types vs expected Mono types:
- If function expects !eco.value and we have primitive → box
- If function expects primitive and we have !eco.value → unbox
- Otherwise → pass through

**ASSUMES**: Signature reflects correct ABI types (monoTypeToAbi).

## Critical Failure Modes

### 1. **Missing/Wrong FuncSignature (MONO_005, CGEN_011)**
```
Mono.MFunction _ _ with no Dict entry
→ Utils.Crash at line 425-428
```

### 2. **Closure Param Count Mismatch (GOPT_001)**
```
closureInfo.params length ≠ stageParamTypes(monoType)
→ Function called with wrong arity
→ Stack corruption at runtime
```

### 3. **Wrong Unboxed Bitmap (CGEN_003, CGEN_026)**
```
Bitmap includes i1 (Bool) as unboxed
→ Runtime GC misinterprets Bool as pointer
→ GC crash or data corruption
```

### 4. **CallInfo.initialRemaining Wrong (CGEN_052)**
```
initialRemaining ≠ actual closure's remaining arity
→ papExtend.remaining_arity attribute wrong
→ Runtime partial/saturated detection fails
→ Wrong number of args consumed
```

### 5. **CallInfo.isSingleStageSaturated Inconsistent**
```
Set to true but args don't saturate all stages
→ generateSaturatedCall called on partial application
→ Wrong call generated (missing papExtend ops)
```

### 6. **CtorShape/CtorLayout Malformed (CGEN_020, MONO_013)**
```
ctorShape.fieldTypes wrong or bitmap wrong
→ eco.construct.custom with wrong attributes
→ Runtime heap allocation with wrong bitmap
→ GC corruption when traversing heap
```

### 7. **MonoType Still Has MVar CNumber (MONO_002)**
```
MVar with CNumber at codegen time
→ monoTypeToAbi returns !eco.value (safe fallback)
→ But indicates failed specialization (should crash earlier)
```

## Key Invariants to Verify

When checking if GlobalOpt/Mono data is correct for MLIR:

1. ✓ Every function-typed node is MonoDefine with MonoClosure OR MonoTailFunc
2. ✓ Every closure has closureInfo.params length == stage arity of its monoType
3. ✓ All CallInfo.initialRemaining values match closure arities at call sites
4. ✓ CallInfo.isSingleStageSaturated is consistent with argument count
5. ✓ All CtorShapes have correct field counts and unboxed bitmaps
6. ✓ No Bool values marked as unboxed in ClosureInfo captures or Constructors
7. ✓ All SpecIds referenced in MonoCall exist in registry
8. ✓ All MonoTypes are fully monomorphic (no unresolved MVars except CEcoValue)
9. ✓ All MonoClosure captures evaluate to valid SSA types
10. ✓ All papExtend.remaining_arity attributes are pre-computed correctly
